## Passos manuais seus (autor-only — agente lista, você faz; mesma divisão do T10 do v0.8):

1. Ativar Apple Developer Program se ainda não estiver.
2. Criar cert **Developer ID Application** (Xcode → Settings → Accounts → Manage Certificates, OU developer.apple.com → Certificates).
3. Exportar do Keychain como `.p12` com senha (Keychain Access → File → Export Items).
4. `base64 -i cert.p12 -o cert.p12.b64`.
5. App Store Connect → Users and Access → Integrations → **App Store Connect API** → criar key role **"Developer"** → baixar `.p8` (UMA vez só) + anotar **Key ID** + **Issuer ID** que o ASC mostra.
6. `base64 -i AuthKey_*.p8 -o key.p8.b64`.
7. Adicionar **5 secrets novos** no GitHub (`SPARKLE_PRIVATE_KEY` do v0.8 continua, NÃO recriar):
   - `DEVELOPER_ID_P12_BASE64` — conteúdo do `.p12.b64`
   - `DEVELOPER_ID_P12_PASSWORD` — senha que usou no export
   - `APPSTORE_CONNECT_API_KEY_P8` — conteúdo do `key.p8.b64`
   - `APPSTORE_CONNECT_KEY_ID` — string ~10 chars
   - `APPSTORE_CONNECT_ISSUER_ID` — UUID

`gh secret set X < arquivo` (com `gh auth switch --user AlexandreAkao` antes — dual-account).

## Mudanças no `release.yml` (NÃO é brainstorm — segue o que já existe):

Substitui o step "Ad-hoc sign with hardened runtime" do v0.8 por uma sequência de 6 novos steps, mantendo intactos os steps já existentes (checkout, select Xcode 16.2, install XcodeGen, generate, read version, guard placeholder, build unsigned, download Sparkle tools, EdDSA-sign, gh release create, update appcast on main). Em ordem:

1. **Import Developer ID cert** — decodar `$DEVELOPER_ID_P12_BASE64` → `.p12` temp; `security create-keychain` keychain temporária; `security set-keychain-settings` (sem lock); `security import` o .p12 com a senha; `security set-key-partition-list -S apple-tool:,apple:` (libera codesign sem prompt); `security list-keychain -s` (adicionar à lista de busca).
2. **Sign com Developer ID Application** — `codesign --force --deep --options runtime --timestamp --sign "Developer ID Application: <Team Name> (TEAMID)" "$APP"`. (Identidade pode ser via hash do cert pra ser estável; ver `security find-identity -v -p codesigning`.) **`--timestamp` é OBRIGATÓRIO** — sem ele a notarização rejeita.
3. **Verify** — `codesign --verify --deep --strict --verbose=2 "$APP"`.
4. **Zip pro notarytool** — `ditto -c -k --keepParent "$APP" "DevTray-${VER}-notary.zip"`.
5. **Notarize** — `xcrun notarytool submit "DevTray-${VER}-notary.zip" --key /tmp/key.p8 --key-id "$KEY_ID" --issuer "$ISSUER_ID" --wait`. Bloqueia 5–15min típico (pode ir a 1h+). Se status != "Accepted" → workflow falha + chamar `notarytool log <id>` pra trazer o motivo (servirá pra debug).
6. **Staple** — `xcrun stapler staple "$APP"`. Grava o ticket dentro do bundle (Gatekeeper aceita sem rede).
7. **Re-zip stapled app** — `ditto -c -k --keepParent "$APP" "DevTray-${VER}.zip"`. O zip do step 4 ficou stale (o stapler escreveu DENTRO do .app, então o zip antigo não tem o ticket). Re-zipar é a forma menos confusa.

EdDSA-sign do zip novo via `sign_update` (igual hoje). `gh release create` + commit do appcast em `main` continuam idênticos. Cleanup no fim do job: `security delete-keychain` da temp + `rm -f /tmp/key.p8 /tmp/cert.p12`.

## Lições que valem carregar (estão na memória, mas reforçando):

- **CI usa Xcode 16.2** (não 15.4 — lesson v0.8). `sudo xcode-select -s /Applications/Xcode_16.2.app`. Brew xcodegen atual emite project format 77 que precisa de Xcode 16+; mantém alinhado.
- **Scheme é `DevTrayApp`, NÃO `DevTray`** (`xcodebuild -project DevTray.xcodeproj -scheme DevTrayApp ...`). `-scheme DevTray` falha com cascade enganoso de "unable to resolve module dependency: DevTrayCore".
- **O app bundle se chama `DevTrayApp.app`** (segue o nome do target; CFBundleDisplayName "DevTray" é só o que aparece na UI). Lesson dolorosa do primeiro release run do v0.8 — `release.yml` hoje usa esse path, manter.
- **Bump de versão em DOIS lugares** (`project.yml` info.properties + `App/DevTrayApp/Info.plist`). v0.10 → "0.10.0" / build "10".
- **`xcodegen generate` depois de editar `project.yml`**; `.xcodeproj` é gitignored. **NÃO editar `project.pbxproj` na mão** — arquivos novos em `App/DevTrayApp/` são auto-incluídos por `xcodegen generate`.
- **`gh` dual-account:** repo é AlexandreAkao (SSH push funciona sempre); active `gh` costuma ser AlexandreAkiraAkao — `gh auth switch --user AlexandreAkao` antes de qualquer API op (PR, `gh secret set`, `gh release create`), restaurar depois.
- **Padrão de release vigente (v0.8+v0.9):** PR merge ff no `main` + tag anotada manual (`vX.Y.Z`) dispara `release.yml`. Local branch deletada pós-merge, ref remota mantida. Este ciclo segue o mesmo ritual.
- **`SPARKLE_PRIVATE_KEY` secret já está setado** (v0.8). NÃO recria — só adiciona os 5 secrets novos. `SUPublicEDKey` no `Info.plist` também continua o mesmo (`ZxyqDIzwmdH+/nfaxtGzTqG+ZZ3ZgNob8Ei0gS3k948=`).
- **`--timestamp` no codesign é não-negociável** pra notarização. Sem ele, notarytool rejeita com erro genérico.
- **`notarytool log <submission-id>`** traz o detalhe quando o `--wait` reporta failure — incluir esse step no workflow no caminho de erro pra debug não ficar cego.
- **Sparkle 2 code-signing consistency check** (brainstorm #5) — beta testers do v0.9.x precisam redownload manual UMA vez quando o v0.10.0 sair. Notar nos release notes.
- **SwiftFormat lint no CI ativo** (v0.9). Rodar `./scripts/format.sh` antes de commitar mudanças no `release.yml` se SwiftFormat tiver opinião (provavelmente não tem — YAML não passa pelo lint).
- **A tag dispara o release CI** (v0.8 changeu o ritual). Para ESTE ciclo, **NÃO** taggear até os 5 secrets estarem setados E o cert importado funcionar — o guard de placeholder não cobre cert ausente. Workflow vai falhar barulhento no step de import se faltar secret.
- **Notarytool é informativo mas verbose; primeira submissão pode demorar mais** que as subsequentes da mesma conta. Não interpreta lentidão como bug.
