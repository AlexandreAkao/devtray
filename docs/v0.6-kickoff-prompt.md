v0.6 entrega:

1. **Export/import de snippets (JSON) + aba Settings → General.** Esse é o pedaço principal. A spec geral (`docs/superpowers/specs/2026-05-20-devtray-design.md`, seção "Backup / migration between machines") diz: Settings → **Export snippets** escreve um JSON; **Import snippets** lê de volta; é o caminho v1 entre Macs antes de CloudKit. Hoje o `SnippetStore` (DevTrayCore) já tem `all()` (export = serializar) e `save()` upsert-by-id (import = desserializar e salvar cada um). A aba General não existe ainda — o `SettingsView` em `App/DevTrayApp/DevTrayApp.swift` só tem `ShortcutsTab` + `AboutTab`.

2. **Fix do `⌘↩` no Spotlight (`currentSelectedID()` retorna nil).** Carregado desde v0.4. Agora com 8 tools (e Snippets sendo o tool de "navegar sem preload") vale mais. Caminho conhecido: `SpotlightPanel.onReturnWithCommand` recebe o `ToolID` atual via closure parametrizada; o view model expõe `selectedID` pro panel via ref compartilhada. ~30 linhas, mexe na borda SwiftUI↔AppKit que é frágil.

3. **Polish do SnippetEditorView que ficou no v0.5:**
   - **Toggle de favorito no editor** — a spec do v0.5 listava um toggle de favorito no editor; o plano simplificou e só deixou via estrela na lista. Adicionar no editor.
   - **InlineErrorBanner no Snippets** — a spec dizia pra usar `DevTrayUI.InlineErrorBanner`; o `SnippetsModel` hoje guarda `errorMessage: String` e a view mostra `Text` vermelho. `InlineErrorBanner` recebe um `ToolError` — precisa de uma ponte (ou um `ToolError.storageFailure(message:)`, ou trocar o tipo do `errorMessage`).

Antes de escrever spec/plano, faz brainstorm rápido sobre:

1. **Shape do JSON de export — versionado desde já?** A spec lista como open question: "Final snippet export JSON shape (needs to be stable across versions before public release)". Duas opções: (a) envelope versionado `{ "version": 1, "exportedAt": ..., "snippets": [...] }` — permite migração de formato no import futuro, custa um wrapper; (b) array puro `[ {snippet}, ... ]` — mais simples, mas mudar o formato depois quebra compatibilidade. Como vamos shippar público com Sparkle, vale decidir a estabilidade agora. Decide o trade-off antes do plano.

2. **Estratégia de merge no import.** Importar um JSON sobre uma biblioteca existente: (a) **merge by id** (upsert — ids iguais sobrescrevem, novos somam) — usa o `save()` que já existe, sem destruir o que o usuário tem; (b) **replace all** (limpa tudo e importa) — precisa de um `deleteAll()` novo no protocolo + é destrutivo (confirmação?); (c) **skip duplicates** (só insere ids que não existem). Recomendação provável: merge by id (não-destrutivo, reaproveita save). Mas decide — afeta se o `SnippetStore` ganha método novo.

3. **Onde mora a lógica de export/import.** Não é um Tool package novo — é uma feature de Settings que usa o `SnippetStore`. Opções: (a) um `SnippetArchive` (encode/decode JSON do `[Snippet]`) em DevTrayCore, testável puro, + os `NSSavePanel`/`NSOpenPanel` na aba de Settings no app target; (b) tudo no app target. Recomendação: codec puro e testável em DevTrayCore (o `Snippet` precisaria virar `Codable` — hoje não é), file pickers no app. Decide se `Snippet: Codable` entra no DevTrayCore.

4. **`⌘↩` fix — entra no v0.6 ou espera?** É tangencial ao tema "Settings + export/import". Mas é dívida desde v0.4 e o caminho já tá mapeado. Vale acoplar (fecha a dívida do Spotlight de vez) ou manter focado em Settings/export e empurrar de novo? Decide o escopo.

**Decisão de empacotamento (não é pergunta de brainstorm, decide na arquitetura):** export/import NÃO é um Tool package. O codec (`SnippetArchive` ou similar) mora em `DevTrayCore` junto do `Snippet`/`SnippetStore`; os file pickers (`NSSavePanel`/`NSOpenPanel`) e a aba `GeneralTab` moram no app target (`App/DevTrayApp/`), seguindo o padrão de `ShortcutsTab`/`AboutTab` no `SettingsView`. Se o import precisar de `deleteAll()`, adiciona no protocolo `SnippetStore` + nas duas implementações (`SQLiteSnippetStore`, `InMemorySnippetStore`).

**Lições do v0.5 que valem carregar (estão na memória, mas reforçando):**
- **XcodeGen é a fonte da verdade.** `DevTray.xcodeproj` é gitignored e gerado por `xcodegen generate` a partir de `project.yml`. Editar pbxproj na mão não é durável (some no próximo generate). Pra adicionar package/target: edita `project.yml` (+ `scripts/test.sh` se tiver testTarget) e roda `xcodegen generate`. Arquivos `.swift` novos em `App/DevTrayApp/` também exigem regenerate.
- **Bump de versão mora em DOIS lugares:** `project.yml` (CFBundleShortVersionString/CFBundleVersion nas info.properties) E `App/DevTrayApp/Info.plist`. O footer lê via `AppMetadata.version` (`CFBundleShortVersionString`). v0.6 → "0.6.0" / build "6". Faz o bump como parte do branch, antes do PR (no v0.5 ele faltou e o footer ficou mostrando v0.4).
- **`XCTAssert(try await …)` não compila** — async não vai em autoclosure. Quebra em `let` local antes do assert. (Vale baixar isso pro plano: escrever os testes async já com `let` bindings.)
- **`@Environment(\.xxx)` cross-module: SEMPRE com type annotation explícito** (`: any SnippetStore`), senão cold SwiftPM / SourceKit falham na inferência do key path.
- **O `NSHostingView` do Spotlight NÃO injeta `\.usageStore` nem `\.preloadBus`** — só `\.snippetStore` (que adicionamos no v0.5). Se o fix do `⌘↩` ou qualquer env novo precisar chegar nas tool views hosteadas inline, injeta nos DOIS lugares: `PopoverRoot` (body do app) E o `NSHostingView(rootView:)` em `SpotlightController.installContent`.
- **`gh` dual account:** o repo é do `AlexandreAkao` (SSH push funciona sempre). O `gh` ativo costuma ser `AlexandreAkiraAkao` — `gh auth switch --user AlexandreAkao` antes de qualquer op de API (PR), e restaura depois.
- **Padrão de store:** SQLite + `InMemory` fallback no app (`makeXxxStore()`), store concreto injetado via `@Environment` a partir do app target, protocolo em DevTrayCore. Tool/engine packages dependem do protocolo, não do GRDB.
- **FTS sync via GRDB `synchronize(withTable:)`** (não triggers manuais); upsert exclui `created_at`. (Relevante se o import mexer em snippets — o `save()` upsert já preserva `created_at`, então import por id mantém a data de criação original.)
- **Manual smoke test depois de cross-module/SwiftPM: Clean Build Folder (`⌥⇧⌘K`) antes do primeiro run.** SourceKit cross-module lag é ruído normal — confia em `swift test` + `xcodebuild`.
