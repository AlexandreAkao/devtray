Quero começar o v0.5 do DevTray. v0.4 acabou de shippar (memória `devtray-state.md` tem o estado atual; tag `v0.4.0` no `main`, ~175 testes em 10 packages, Spotlight panel ativo com `⌃⌥Space` default, smart-paste por tool, Plano B em vigor — Spotlight hosta tools inline em vez de abrir o popover).

v0.5 adiciona o **`SnippetsTool`** — primeiro tool com persistência heavy. Inclui schema `snippets` + `snippets_fts` (FTS5), protocolo `SnippetStore` em `DevTrayCore`, `SQLiteSnippetStore` em `DevTrayStorage`, UI de lista + editor + favoritos + busca full-text. Veja spec geral em `docs/superpowers/specs/2026-05-20-devtray-design.md` seções "Release phasing → v0.5", "Data and persistence → Schema (snippets, snippets_fts)" e "UI surfaces → Tool views" pro padrão de tool com store. Template do plano é o v0.4 em `docs/superpowers/plans/2026-05-23-devtray-v0.4-spotlight.md` (mais recente, inclui o padrão de Plano A/B e fix-no-meio-do-plano).

Antes de escrever spec/plano, faz brainstorm rápido sobre:

1. **SwiftPM stub workaround — finalmente resolve (gotcha #1 carregado desde v0.2).** `SnippetsTool` é o primeiro novo Tool package com engine + view desde v0.2. Volta a dor: engine task no plano roda antes da view task, `Sources/SnippetsTool/` fica vazio, `swift test` falha com "Source files for target SnippetsTool should be located under Sources/SnippetsTool/". Três caminhos: (a) baked stub na instrução do engine task — engine cria `Sources/SnippetsTool/SnippetsTool.swift` com 2 linhas de placeholder que a view task sobrescreve; (b) split do `Package.swift` em duas tasks — primeira declara só `SnippetsToolKit`, segunda adiciona `SnippetsTool` quando já tem fonte; (c) inverter ordem do plano — view-side primeiro (com stub do engine), engine depois. Decide o trade-off antes de escrever o plano.

2. **Tags do snippet — coluna JSON ou junction table?** Spec diz `tags TEXT NOT NULL DEFAULT '[]'` (JSON array inline na linha). Mais simples, indexável via FTS, sem joins. Junction table (`snippet_tags(snippet_id, tag)`) seria correto relacionalmente e permite "todos snippets com tag X" mais rápido, mas exige 2 tabelas a mais (junction + tags_unique) e migrations mais barulhentas. Pra v0.5 — JSON inline tá no spec, mas vale confirmar dado que tags são feature central.

3. **FTS5 — triggers SQL manuais ou helper do GRDB?** GRDB tem `Database.create(virtualTable:)` + observação automática, mas a sync FTS↔snippets exige 3 triggers (INSERT, UPDATE, DELETE) que tradicionalmente são SQL puro. Manual = controle total, mais legível, mais código pra testar. GRDB helper = menos código mas perde-se em transparência. Spec não decidiu.

4. **⌘↩ no Spotlight — fix agora?** v0.4 deixou como known-limitation (`currentSelectedID()` retorna nil). Snippets é o tool com maior valor pra "navegar sem preload" (abrir lista sem pré-colar clipboard). Caminho do fix: `SpotlightPanel.onReturnWithCommand` passa o `ToolID` via closure parametrizada, view model expõe `selectedID` pro panel via ref compartilhada. ~30 linhas, mas mexe na borda SwiftUI↔AppKit que é frágil. Vale acoplar ao v0.5 (faz sentido com Snippets) ou esperar v0.6+ quando tiver mais tools?

**Decisão de empacotamento:** `SnippetsTool` segue o padrão dos outros 7 Tool packages — `Packages/Tools/SnippetsTool/` com `SnippetsToolKit` (engine, depende só de DevTrayCore + DevTrayStorage) e `SnippetsTool` (view, depende do Kit + DevTrayUI). O `SQLiteSnippetStore` mora no `DevTrayStorage` pra ficar junto do `SQLiteUsageStore` (e compartilhar o `DatabaseOpener` + `DatabaseQueue`), o protocolo `SnippetStore` em `DevTrayCore`. Decide isso na arquitetura, não como pergunta de brainstorm.

**Decisão de schema migration:** v0.5 adiciona `v2_snippets` (cria `snippets`, `snippets_fts`, triggers, índices) no `Migrations.swift` existente. Não toca em `tool_usage`. Test fixtures pra v1→v2 incluem pelo menos um snippet pré-inserido pra confirmar que migration não corrompe dados.

**Lições do v0.4 que vale carregar:**
- Se um método precisa ser observável via generic constraint (`<T: Tool>`), declara no protocol body, NÃO só em extension. Extension-only = static dispatch = overrides ignorados. O `clipboardMatch` queimou 7 of 10 testes do ranker até o bug ser caçado.
- `@Environment(\.xxx)` cross-module: SEMPRE com type annotation explícito (`: PreloadBus`), senão SourceKit / cold SwiftPM falham na inferência de key path.
- Atalhos globais: checa colisão com macOS antes de cravar o default. `⌥⌘Space` colide com Finder Search. `⌃⌥Space` é livre. Tem lista oficial em System Settings → Keyboard → Keyboard Shortcuts.
- `.onKeyPress(...)` em SwiftUI só dispara na view focada. Pra teclas que precisam funcionar com foco no TextField (esc, ↑/↓), ou anexa direto na TextField OU override `cancelOperation`/`keyDown` no `NSPanel` (responder chain). `cancelOperation` é o canônico pra esc.
- `MenuBarExtra` no macOS 14 não tem API pública pra abrir programaticamente. KVC walk em `NSStatusItem.button.performClick` NÃO funciona. Se um futuro caminho precisar disso (ex.: notificação clica e abre popover), planeja Plano B desde o começo.
- Manual smoke test depois de cross-module work em SwiftPM: SEMPRE Clean Build Folder (`⌥⇧⌘K`) antes do primeiro run. Incremental build engole edits em `Packages/*/Sources/`.
- SourceKit cross-module lag continua normal — confia em `swift test` + `xcodebuild`, ignora warnings transientes do editor.
