## Antes de escrever spec/plano, faz brainstorm rápido sobre:

1. **Dependências externas vs engine hand-rolled.** O `yaml` praticamente exige um parser de YAML (hand-rolling YAML é inviável) — provável **Yams** (lib SwiftPM padrão). Os outros 4 dá pra fazer com Foundation/stdlib: regex (`NSRegularExpression` ou Swift `Regex`), diff (`CollectionDifference` ou Myers à mão), color (matemática HEX/RGB/HSL + `ColorPicker` do SwiftUI), cron (parser próprio, é bounded). **Decide: o v0.7 introduz dep externa (Yams)?** Isso mexe em `Package.resolved` (lembrar do churn que vaza pro primeiro leaf package — ver gotcha do v0.4) e no build offline. Recomendação provável: Yams só pro yaml, resto hand-rolled.

2. **Engine de regex: Swift `Regex` vs `NSRegularExpression`.** Swift Regex é mais moderno mas tem peculiaridades; `NSRegularExpression` é batido e entrega grupos por `NSRange` fácil. Afeta a API do `RegexEngine` e os testes. Decide.

3. **Diff: granularidade + algoritmo.** Line-level vs char/word-level; `CollectionDifference` (stdlib, simples) vs Myers próprio (controle de hunks pra unified). E qual a view primária — side-by-side ou unified. Decide o suficiente pra spec.

4. **Quais tools ganham `clipboardMatch` (smart-paste no Spotlight) e em qual confidence.** Candidatos: `color` (hex tipo `#RRGGBB` → strong), `cron` (5 campos cron → strong/weak), `yaml` (parece YAML → **weak e arriscado**, colide com JSON do tool `json`), `regex`/`diff` (sem sinal de clipboard útil → sem matcher). Decide a lista de matchers — lembra que `clipboardMatch` JÁ é requisito do protocolo `Tool` (dispatch por witness table, ver lição do v0.4), e o matcher mora em `XxxToolKit/XxxClipboardMatcher.swift` com extension de uma linha no `XxxTool`.

5. **Tamanho do batch.** 5 tools é o maior batch de tools num ciclo só (v0.2 fez 4). Se na hora do plano parecer grande demais, dá pra fechar 3 (regex+diff+color) e empurrar cron+yaml — mas como cada tool é uma fatia independente, provavelmente cabe num plano só com tasks bem isoladas. Decide se mantém os 5.

## Decisões de arquitetura/empacotamento (NÃO é pergunta de brainstorm — segue o padrão consolidado dos 8 tools):

Cada tool é um **Swift package** em `Packages/Tools/XxxTool/` com dois products:
- **`XxxToolKit`** — engine puro, testável, depende **só de DevTrayCore** (não DevTrayUI, não DevTrayStorage). Contém `XxxEngine` (Result<_, ToolError>) + `XxxClipboardMatcher` (se o tool tiver smart-paste).
- **`XxxTool`** — view SwiftUI, depende de `XxxToolKit` + DevTrayCore + DevTrayUI. Contém a `enum XxxTool: Tool` (conformance: `id`, `displayName`, `iconName`, `keywords`, `category`, `makeView()`, e o override de `clipboardMatch` via extension de uma linha chamando o matcher) + a `XxxToolView`.
- `testTarget` `XxxToolKitTests`.

Modelo a copiar: **`Packages/Tools/URLTool`** é o mais simples (engine + matcher + view). Olhar também como `URLToolView` consome `@Environment(\.preloadBus)` + `applyPendingPreloadIfMatches()` pro smart-paste — **e nota que o v0.6 finalmente injetou `\.preloadBus` no `NSHostingView` do Spotlight**, então as views novas recebem preload tanto no popover quanto inline no Spotlight de graça (não precisa fazer nada extra além de ler o env, com type annotation explícito).

**Registro de cada tool novo (4 lugares):**
1. `App/DevTrayApp/DevTrayApp.swift` → `import XxxTool` + `r.register(XxxTool.self)` em `makeRegistry()`.
2. `project.yml` → bloco `packages:` (`XxxTool: path: Packages/Tools/XxxTool`) **E** `targets.DevTrayApp.dependencies:` (`- package: XxxTool`).
3. `scripts/test.sh` → array `PACKAGES` (senão os testes do tool não rodam).
4. Rodar **`xcodegen generate`** depois de mexer no `project.yml` (o `.xcodeproj` é gitignored e gerado).

## Lições que valem carregar (estão na memória, mas reforçando):

- **⚠️ O scheme do Xcode é `DevTrayApp`, NÃO `DevTray`.** `xcodebuild -project DevTray.xcodeproj -scheme DevTrayApp build`. Usar `-scheme DevTray` falha com "does not contain a scheme named DevTray" / um cascade enganoso de "unable to resolve module dependency: DevTrayCore". (O plano do v0.6 errou isso o tempo todo — não repetir.)
- **`clipboardMatch` TEM que estar no corpo do protocolo `Tool`** (já está), senão generic callers usam static dispatch e os matchers viram dead code (bug latente que pegou no v0.4). O matcher concreto fica em `XxxToolKit/XxxClipboardMatcher.swift`.
- **Gotcha SwiftPM target-sem-sources** (carregado desde v0.2): o task do engine roda antes do task que cria `Sources/XxxTool/`, e o SwiftPM rejeita target sem sources ("target referenced in product is empty"). Bake um stub de 2 linhas em `Sources/XxxTool/` no task do package (overwritten depois pela view), OU declara o target da view no `Package.swift` só depois das sources existirem. O v0.5 resolveu com o stub bakeado — funcionou.
- **`@Environment(\.xxx)` cross-module: SEMPRE com type annotation explícito** (`: PreloadBus`, etc.), senão SwiftPM frio / SourceKit falham na inferência do key path.
- **`XCTAssert(try await …)` não compila** — async não vai em autoclosure. Quebra em `let` local antes do assert. (Já escrever os testes de engine async com `let` bindings se houver — mas a maioria dos engines aqui é síncrona.)
- **Bump de versão mora em DOIS lugares:** `project.yml` (info.properties) E `App/DevTrayApp/Info.plist`. v0.7 → "0.7.0" / build "7". Faz o bump como parte do branch, antes do PR (o footer lê via `AppMetadata.version`).
- **Se entrar dep externa (Yams):** declara no `project.yml` (`packages:` com `url:`/`from:` igual o KeyboardShortcuts) + linka no `DevTrayApp` se necessário; o `YamlToolKit` declara a dep no seu `Package.swift`. Esperar **churn no `Package.resolved`** (vaza pro primeiro leaf package porque o workspace não tem root `Package.swift` buildado) — commitar pra reprodutibilidade, igual fizemos com KeyboardShortcuts no v0.4.
- **SourceKit cross-module lag é ruído normal** — confia em `swift test` (por package) + `./scripts/test.sh` (suite toda) + `xcodebuild -scheme DevTrayApp`. **Clean Build Folder (`⌥⇧⌘K`) antes do primeiro smoke manual.**
- **Lição forte do v0.6 — seams de injeção de ambiente:** toda Scene que consome um `@Environment` store precisa injetar ele (o bug crítico do v0.6 foi a cena `Settings` não injetar `\.snippetStore`). Pros tools novos isso não deve aparecer (eles só leem `preloadBus`, já injetado nos dois lugares), mas se algum tool precisar de um env novo, lembra: injeta no `PopoverRoot` E no `NSHostingView` do `SpotlightController`.
