## Lições que valem carregar (do v1.0 + v1.0.1 post-mortems):

- **Live docs check é OBRIGATÓRIO antes do spec.** Cada bug do v1.0 cycle veio de assumir Paddle Classic mental model do training data. Pra QUALQUER API externa: fetch live docs DURANTE brainstorm, não depois do bug em prod.
- **Money flows precisam de polling + webhook (defense in depth).** Webhooks são unreliable (CF blip, Paddle não dispatch, network). Para qualquer state crítico, polling reconciliation é parte da arquitetura desde dia 1. v1.0.1 provou isso end-to-end.
- **Status fields matter** — Paddle Billing v1 tem `status: pending_approval` em refunds live; revogar antes de `approved` é money-out incorrect. Sempre confirmar status semantics nos docs.
- **`scripts/format.sh` mutate files outside task scope** — v1.0 PR #10 quase quebrou CI lint porque T8/T9 rodaram format.sh que corrigiu `LicenseClient.swift`, mas targeted `git add` não pegou. Recovery: format.sh dá `git add -u` automatic OR implementer prompt requer `git status` antes do commit.
- **API key scopes têm que casar com TODOS os endpoints chamados** — v1.0 instructions só pediam `transactions:write + prices:read`; v1.0.1 adicionou `transactions:read + customers:read`. Sempre listar TODOS scopes que o handler usa no comment do wrangler.toml.
- **`wrangler deployments status` NÃO mostra cron triggers** — pra verificar use o dashboard (Workers & Pages → Worker → Triggers tab) OR procura a linha `schedule:` no `wrangler deploy` stdout.
- **workers.dev subdomain é pre-req pra cron triggers** — mesmo se você só usa custom domain. Account-level setup, criado automatic ao abrir Workers & Pages no dashboard 1ª vez.
- **CF KV `list()` retorna até 1000 keys sem pagination** — `list.list_complete === false` indica truncation. v1.0+v1.0.1 warned mas não pagina. Add cursor-based pagination quando license count se aproxima de 1000.
- **`wrangler kv key get/put` aceita `--namespace-id` ou `--binding`** — usar ID direto evita conflito com env config.
- **Padrão de release (v0.8/v0.9/v0.10/v0.11/v1.0/v1.0.1):** PR merge no main via `gh pr merge --rebase` (NÃO `--merge`) + tag anotada manual (`vX.Y.Z`) dispara `release.yml`. Pipeline warm = ~3 min.
- **`gh` dual-account:** repo é AlexandreAkao (SSH push sempre); `gh` ativa em AlexandreAkao por padrão agora (confirmado v1.0+v1.0.1).
- **`SPARKLE_PRIVATE_KEY` + `SUPublicEDKey` continuam intactos.** **DO NOT recriar.** License keypair também NÃO recria — está safe em CF Workers secret + `~/devtray-secrets/license_private.b64`.
- **Bundle ID é `com.devtray.app`** (NÃO `com.devtray.DevTrayApp` — esse último é só o URL scheme name no CFBundleURLName).
- **File on disk é `DevTray.app`** (PRODUCT_NAME=DevTray; APP_PATH no release.yml resolve `build/Build/Products/Release/DevTray.app`).
- **Notarytool warm** — release.yml runs após v0.10 (cold 41min) ficam ~2-3min.
- **SwiftFormat lint no CI ativo** — `./scripts/format.sh` antes de commit Swift.
