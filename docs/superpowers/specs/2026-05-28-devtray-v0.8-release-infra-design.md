# DevTray v0.8 — Release / Distribution infra (Sparkle + release CI, ad-hoc beta) · Design

**Date:** 2026-05-28
**Cycle:** v0.8 (makes DevTray distributable: in-app auto-update + tag-triggered release pipeline). First infra cycle after the v1 tool surface (13 tools) was completed in v0.7.
**Branch:** `feat/v0.8-release` (in place, no worktree — manual smoke / packaging runs better on the main checkout, same as v0.5/v0.6/v0.7)
**Predecessor:** v0.7 spec `docs/superpowers/specs/2026-05-27-devtray-v0.7-tools-design.md`, plan `docs/superpowers/plans/2026-05-27-devtray-v0.7-tools.md`
**Reference:** general design `docs/superpowers/specs/2026-05-20-devtray-design.md` (Release phasing). Reconciled roadmap: auto-memory `devtray-roadmap.md`.

## Goal

Make DevTray installable and self-updating for people other than the author, **without yet paying for the Apple Developer Program**. Three legs land together, all under **ad-hoc code signing**:

1. **Sparkle** — in-app auto-update (appcast feed + EdDSA-signed updates + a "Check for Updates…" action).
2. **Release CI** — a workflow triggered by a `vX.Y.Z` tag that builds Release, ad-hoc signs with hardened runtime, zips, EdDSA-signs the archive, updates the appcast, and publishes a GitHub Release with the zip asset.
3. **Test CI reconciliation** — `test.yml`'s package matrix is stale (4 of 16 packages). Replace it with a single-source dynamic matrix so it never drifts again.

**Explicitly sliced out of this cycle (the deciding brainstorm call):** the author does **not** have an active Apple Developer Program membership. Developer ID signing + notarization is a **future slice** (likely v0.9). v0.8 ships an **ad-hoc "beta"**: integrity of every update is guaranteed by Sparkle's EdDSA signature, but first launch on someone else's Mac shows Gatekeeper friction (right-click → Open). That friction disappears once Developer ID + notarization land — nothing in v0.8 needs to be redone, only added. Hardened runtime is already `YES`, which keeps the path to notarization clean.

## Brainstorm decisions

| Question | Decision | Why |
|---|---|---|
| Apple Developer account / distribution method | **Ad-hoc beta now; Developer ID + notarization later.** Direct distribution (GitHub Releases), **not** Mac App Store. | Author has no paid membership yet. Direct-distribution (NotchNook model) keeps the app **unsandboxed** so `NSOpenPanel`/`NSSavePanel` (snippet export/import), the `KeyboardShortcuts` global hotkey, and Sparkle all keep working untouched — MAS would force sandboxing + entitlements and forbids Sparkle. EdDSA secures updates regardless of code-signing, so auto-update works ad-hoc. |
| Packaging format | **Zip now; DMG at v1.0 launch.** | Sparkle accepts both. Zip keeps the CI simple (`ditto` + `sign_update`, no `create-dmg` step to get right). The "drag to /Applications" DMG experience is a launch-polish concern, not a beta concern. |
| Appcast hosting | **GitHub Pages from `/docs`** → `https://alexandreakao.github.io/devtray/appcast.xml` | Stable, versioned, fixed `SUFeedURL`. CI commits the updated `docs/appcast.xml` each release. `raw.githubusercontent` has aggressive caching/poor headers; a release asset makes a fixed feed URL fragile. |
| `test.yml` reconciliation | **Single-source dynamic matrix.** A `discover` job emits the package list (same source as `scripts/test.sh`); the test job fans out via `fromJSON`. | Covers all 16 packages, keeps per-package parallelism + red/green, and **never drifts again** (the drift is exactly what caused this debt). Hardcoding 16 re-drifts on the next package; a single `test.sh` step loses parallelism/visibility. |
| Tag flow | **Manual annotated tag stays; the tag now triggers the release CI.** | Keeps the existing ritual (merge ff to `main` + author creates `vX.Y.Z`) — author still controls *when* a release publishes. The tag push is simply what fires build → sign → package → publish. |
| Version / numbering | **v0.8 = infra (ad-hoc beta) → `0.8.0` / build `8`.** | Slicing is confirmed; infra is its own cycle. Developer ID/notarization = next slice; launch (README/landing/icon/DMG) = v1.0. |

## Architecture / packaging decisions (not brainstorm — follows what exists)

- **Sparkle dependency** declared in `project.yml` `packages:` (`url: https://github.com/sparkle-project/Sparkle`, `from: 2.x`), linked into the `DevTrayApp` target only — same shape as the existing `KeyboardShortcuts`/`Yams` entries. **Expect `Package.resolved` churn** in the first leaf package (the v0.4/v0.7 gotcha) — commit it.
- **Local dev build stays ad-hoc Automatic, unchanged.** No `DEVELOPMENT_TEAM`, no `CODE_SIGN_IDENTITY` change in `project.yml`. Release signing is handled entirely in the CI build step (override flags there), so a clean local build keeps working exactly as today.
- **No entitlement changes.** The app stays unsandboxed with only `com.apple.security.network.client`. Sparkle on a non-sandboxed app needs no extra entitlements; ad-hoc `--deep` signing covers Sparkle's embedded framework + XPC services.
- **Run `xcodegen generate`** after editing `project.yml` (the `.xcodeproj` is gitignored and generated by XcodeGen).
- **Version lives in two places** — `project.yml` (`info.properties`) **and** `App/DevTrayApp/Info.plist`. Bump both to `0.8.0` / `8`. `AppMetadata.version` reads `CFBundleShortVersionString`.

## Components

### 1. Sparkle integration (app target)

**Info.plist keys** (`App/DevTrayApp/Info.plist`, mirrored as needed in `project.yml` `info.properties`):

| Key | Value |
|---|---|
| `SUFeedURL` | `https://alexandreakao.github.io/devtray/appcast.xml` |
| `SUPublicEDKey` | the EdDSA public key string from `generate_keys` (author provides; scaffolded with a `PLACEHOLDER` until then) |
| `SUScheduledCheckInterval` | `604800` (weekly) |

`SUEnableAutomaticChecks` is **left unset** on purpose → Sparkle shows its standard "check automatically?" permission prompt on first launch (privacy-friendly default).

**Updater wiring** — a small `ObservableObject` in the app target wrapping Sparkle's `SPUStandardUpdaterController`:

```swift
import Sparkle

@MainActor
final class UpdaterController: ObservableObject {
    private let controller: SPUStandardUpdaterController
    @Published var canCheckForUpdates = false

    init() {
        controller = SPUStandardUpdaterController(
            startingUpdater: true, updaterDelegate: nil, userDriverDelegate: nil)
        controller.updater.publisher(for: \.canCheckForUpdates)
            .assign(to: &$canCheckForUpdates)
    }

    func checkForUpdates() { controller.updater.checkForUpdates() }
}
```

Held as a `@StateObject` in `DevTrayApp`. `startingUpdater: true` runs the scheduled background checks.

**UI surfaces for "Check for Updates…":**
- **About tab** (`SettingsView` → `AboutTab` in `DevTrayApp.swift`, app target — can import Sparkle freely): a `Button("Check for Updates…") { updater.checkForUpdates() }.disabled(!updater.canCheckForUpdates)` below the version line. This is the canonical, required surface.
- **Popover footer** (`PopoverRoot` in **DevTrayUI**): to keep DevTrayUI free of any Sparkle import, `PopoverRoot` gains an optional injected closure (`onCheckForUpdates: (() -> Void)? = nil`) + a `canCheckForUpdates: Bool = false`; the app passes `updater.checkForUpdates`. Footer item renders only when the closure is present. No Sparkle dependency leaks into the UI package.

### 2. Release CI (`.github/workflows/release.yml`)

**Trigger:** `push` of tags matching `v*`. `permissions: contents: write` (to create the Release and commit the appcast).

**Single job (macos-14, Xcode 15.4):**
1. `actions/checkout@v4` (full history; we commit back).
2. `sudo xcode-select -s /Applications/Xcode_15.4.app`.
3. `brew install xcodegen`; `xcodegen generate`.
4. **Build Release unsigned, then ad-hoc sign with hardened runtime** (avoids needing a signing identity in CI):
   - `xcodebuild -project DevTray.xcodeproj -scheme DevTrayApp -configuration Release -derivedDataPath build build CODE_SIGNING_ALLOWED=NO`
   - locate `build/Build/Products/Release/DevTray.app`
   - `codesign --force --deep --options runtime --sign - "DevTray.app"` (ad-hoc `-`, hardened runtime preserved)
5. **Zip:** `ditto -c -k --keepParent "DevTray.app" "DevTray-<shortVersion>.zip"`.
6. **EdDSA-sign the archive:** download Sparkle's tools for the pinned version, run `sign_update DevTray-<shortVersion>.zip` with the private key from the `SPARKLE_PRIVATE_KEY` secret (written to a temp keyfile, passed via `-f`). Capture `sparkle:edSignature` + `length` from its output.
7. **Update the appcast:** `scripts/appcast.sh` appends a new `<item>` (see §3) to `docs/appcast.xml` using version, build, pubDate, the release asset download URL, the signature, and the length.
8. **Commit `docs/appcast.xml`** back to `main` (git config bot user; commit + push). GitHub Pages serves it.
9. **Publish the Release:** `gh release create v<shortVersion> DevTray-<shortVersion>.zip` (title = version, generated notes). `GITHUB_TOKEN` from the workflow.

**Version source:** read `CFBundleShortVersionString` / `CFBundleVersion` from `App/DevTrayApp/Info.plist` so the artifact name, appcast item, and release tag stay consistent with a single source. (Sanity-check the pushed tag matches.)

**Secret needed this cycle:** only `SPARKLE_PRIVATE_KEY`. Cert/notary secrets belong to the future Developer ID slice.

### 3. Appcast & GitHub Pages (`docs/appcast.xml`)

- **Skeleton committed in v0.8** — a valid empty channel so Pages serves something before the first CI release:

```xml
<?xml version="1.0" standalone="yes"?>
<rss version="2.0" xmlns:sparkle="http://www.andymatuschak.org/xml-namespaces/sparkle" xmlns:dc="http://purl.org/dc/elements/1.1/">
  <channel>
    <title>DevTray</title>
    <link>https://alexandreakao.github.io/devtray/appcast.xml</link>
    <description>DevTray release feed</description>
    <language>en</language>
  </channel>
</rss>
```

- **Item shape** (appended by `scripts/appcast.sh`, history preserved — newest first):

```xml
<item>
  <title>0.8.0</title>
  <sparkle:version>8</sparkle:version>
  <sparkle:shortVersionString>0.8.0</sparkle:shortVersionString>
  <sparkle:minimumSystemVersion>14.0</sparkle:minimumSystemVersion>
  <pubDate>Wed, 28 May 2026 12:00:00 +0000</pubDate>
  <enclosure
    url="https://github.com/AlexandreAkao/devtray/releases/download/v0.8.0/DevTray-0.8.0.zip"
    sparkle:edSignature="<from sign_update>"
    length="<bytes>"
    type="application/octet-stream"/>
</item>
```

- **Single "stable" channel** for the beta (no separate channel attribute).
- **Why `sign_update` + a template script** instead of `generate_appcast`: `generate_appcast` regenerates the whole feed from a directory of archives, which CI doesn't keep; templating one item from `sign_update` preserves version history cleanly.
- **GitHub Pages** source = `main` / `/docs` — a **manual one-time setup** by the author (repo Settings → Pages). Confirm the exact lowercase owner casing in the served URL matches `SUFeedURL`.

### 4. Test CI reconciliation (`.github/workflows/test.yml`)

Replace the hardcoded 4-package matrix with a **single-source dynamic matrix**:

- Add a `--list` mode to `scripts/test.sh` that prints, as a JSON array, the packages from its `PACKAGES` array **that have a test target** (same `grep testTarget Package.swift` filter the run path already applies). This keeps one source of truth for "what gets tested."
- `discover` job: checkout → `echo "packages=$(./scripts/test.sh --list)" >> "$GITHUB_OUTPUT"`.
- `packages` job: `needs: discover`, `strategy.matrix.package: ${{ fromJSON(needs.discover.outputs.packages) }}`, then the existing per-package `swift test --parallel` step.
- The `app` job (`xcodebuild test -scheme DevTrayApp`) is unchanged.

## Manual (author-only) vs automatable/verifiable split

The kickoff asked for this explicitly — some steps require the author's machine/account and cannot be run by an agent or in CI from inside this cycle:

**Author-only (machine/account):**
1. Run Sparkle's `generate_keys` → private key lands in the login Keychain; provide the **public** key string for `SUPublicEDKey`.
2. Export the private key and add it as the **`SPARKLE_PRIVATE_KEY`** GitHub Actions secret.
3. Enable **GitHub Pages** (repo Settings → Pages → source `main` / `/docs`).
4. The first real tag push (to exercise the release workflow end-to-end) and the first manual smoke.

**Automatable / locally verifiable (agent or CI):**
- Add Sparkle to `project.yml`; `xcodegen generate`; build; wire `UpdaterController` + the two "Check for Updates…" surfaces.
- Add Info.plist keys (with a `PLACEHOLDER` `SUPublicEDKey` until step 1).
- Write `.github/workflows/release.yml`, `scripts/appcast.sh`, the `docs/appcast.xml` skeleton.
- Convert `test.yml` to the dynamic matrix; add `--list` to `test.sh`.
- Bump version to `0.8.0` / `8` in both places; commit `Package.resolved` churn.

## Testing

- **`scripts/test.sh --list`** — verify it emits a valid JSON array of exactly the test-bearing packages (locally: pipe to `jq .`).
- **`scripts/appcast.sh`** — unit-ish check: run against the skeleton with sample inputs; assert a well-formed `<item>` is inserted (newest-first) and the XML still parses (`xmllint --noout`).
- **Full suite:** `./scripts/test.sh` (unchanged behavior; new tools already covered).
- **App build:** `xcodebuild -project DevTray.xcodeproj -scheme DevTrayApp build` — **scheme is `DevTrayApp`, not `DevTray`** (using `-scheme DevTray` fails with a misleading "unable to resolve module dependency: DevTrayCore" cascade).
- **Manual smoke** (author): Clean Build Folder (`⌥⇧⌘K`) first; launch; confirm "Check for Updates…" appears in About + popover footer and is enabled; trigger a check against the live appcast and confirm Sparkle reaches the feed (a "you're up to date" / "update available" dialog — not a network error).
- **Release dry-run:** push a throwaway tag (or `act`-style local reasoning) is out of CI's reach for the agent; the agent verifies the workflow YAML parses and the steps are individually sound. The true end-to-end first release is an author step (it needs the secret + Pages).

## Carried lessons relevant here

- **Scheme is `DevTrayApp`, not `DevTray`** — `xcodebuild -project DevTray.xcodeproj -scheme DevTrayApp …`.
- **Version bump lives in two places** (`project.yml` + `Info.plist`).
- **External dep → `Package.resolved` churn** in the first leaf package; commit it (Yams/v0.7, KeyboardShortcuts/v0.4 precedent).
- **Run `xcodegen generate`** after touching `project.yml`; the `.xcodeproj` is gitignored.
- **`gh` dual account:** repo is `AlexandreAkao` (SSH push works); active `gh` is usually `AlexandreAkiraAkao` — `gh auth switch --user AlexandreAkao` before any API op (creating a Release via `gh`, the PR), restore after. **Note:** the *release CI* uses the in-runner `GITHUB_TOKEN`, so the dual-account dance only applies to local `gh` use.
- **Release ritual evolves this cycle:** still merge ff to `main` + author creates the annotated `v0.8.0` tag — but the tag now **triggers** the release workflow (it no longer only labels a commit). Local branch deleted post-merge; remote ref kept.
- **SourceKit cross-module lag is normal noise** — trust `swift test` / `./scripts/test.sh` / `xcodebuild -scheme DevTrayApp`; Clean Build Folder before the first manual smoke.
- **Secrets are never committed** — EdDSA private key goes in the Keychain + GitHub secret only.

## Out of scope

- **Developer ID signing + notarization + stapling** — the next slice (likely v0.9). Removes the Gatekeeper first-launch friction. Adds cert `.p12` + notary credential secrets; no v0.8 work is thrown away.
- **Mac App Store** — rejected for v1 (would force sandboxing + entitlements, breaks the `NSOpenPanel`/global-hotkey/Sparkle flow).
- **DMG packaging** — deferred to v1.0 launch polish; v0.8 ships zip.
- **Launch assets** — README/landing page/app icon/announcement = v1.0, a separate cycle.
- **Multiple update channels** (beta/stable split, `sparkle:channel`) — single stable channel for now.
- **`generate_appcast`-based full-feed regeneration** — using `sign_update` + template instead to preserve history in CI.
