# DevTray v0.10 — Developer ID signing + notarization (kills Gatekeeper friction) · Design

**Date:** 2026-05-29
**Cycle:** v0.10 (replaces v0.8's ad-hoc signing with Developer ID + xcrun notarytool + stapler). Pure delta on the release pipeline — no Swift code changes. Ships as `0.10.0` / build `10`.
**Branch:** `feat/v0.10-notarization` (in place, no worktree — smoke + secret setup run better on the main checkout, same as prior cycles)
**Predecessor:** v0.8 spec `docs/superpowers/specs/2026-05-28-devtray-v0.8-release-infra-design.md` (release infra original — this cycle is the delta on top of it). v0.9 spec `docs/superpowers/specs/2026-05-29-devtray-v0.9-complete-v1-design.md` (most recent format).
**Reference:** roadmap memory `devtray-roadmap.md` (Developer ID + notarization was originally on the v0.9 line; got swapped with the "complete v1 surface" cycle because it depends on the paid Apple Developer Program).

## Goal

The v0.8 release pipeline publishes ad-hoc signed zips. First launch on someone else's Mac shows the **"developer cannot be verified"** Gatekeeper dialog and forces right-click→Open or a trip through Settings → Privacy. Fine for a beta among friends; **fatal for a paid app**. The next cycle (v0.11) is paywall + license validation, and that only makes sense after this friction dies.

This cycle replaces ad-hoc signing with:

1. **Developer ID Application** code signing (real cert, hardened runtime, secure timestamp).
2. **xcrun notarytool** submission (Apple inspects the bundle for malware/notarization rules; returns a ticket).
3. **xcrun stapler** (writes the ticket into the bundle so Gatekeeper accepts it without an internet round-trip).

End result: a downloaded zip from GitHub Releases opens on first launch with no Gatekeeper friction. Sparkle auto-update (EdDSA) keeps working unchanged for users on v0.10+; users still on v0.8/v0.9 (ad-hoc) need to redownload v0.10.0 manually once because Sparkle 2's code-signing consistency check blocks the team-identifier change from "none" → "real".

**No Swift code changes this cycle.** Only `release.yml`, `project.yml`, and `Info.plist`. Hardened runtime was already `YES` on the v0.8 ad-hoc step, so the path was kept clean for exactly this.

## Brainstorm decisions

| Question | Decision | Why |
|---|---|---|
| Apple Developer Program status | **Active, cert not yet exported.** Spec/plan in full; cert export + 5 secrets are author-only execution steps. | The membership is the hard dependency; once it's active the rest of execution is a normal mix of agent YAML work and author-machine steps (same split as v0.8 T10). |
| Notarytool auth | **App Store Connect API key (3 secrets: `.p8` + Key ID + Issuer ID).** | Independent of the personal Apple ID; survives password rotation; modern path Apple endorses. App-specific password rejected (amarra ao Apple ID + quebra na rotação). |
| Packaging | **Zip** (signed + notarized + stapled `.app` inside). | Keeps pipeline simple. DMG deferred to v1.0 launch (needs custom icon + background + branding — packaging polish, not security). |
| Numbering | **`0.10.0` / build `10`** (minor bump). | Distribution-model change (ad-hoc → signed-for-real) is meaningful. Matches v0.4…v0.9 pattern of minor-bumps per cycle. v1.0 reserved for launch. |
| Sparkle compatibility (ad-hoc → notarized) | **Release notes only; no in-app migration helper.** | Sparkle 2 code-signing consistency check blocks the team-identifier transition. Beta-tester universe ≈ 0–handful; a one-time manual redownload note in the v0.10.0 release body is enough. From v0.10 → v0.11 → … auto-update flows clean. |
| Codesign identity selection | **Auto-detect SHA-1 hash via `security find-identity`.** | No 6th secret for team name/ID; works regardless of cert display string; failure mode is loud at the import step. Hardcoding the `"Developer ID Application: Name (TEAMID)"` string was the alternative — more cerimonial, no robustness gain. |

## Architecture / pipeline decisions (not brainstorm — follows what exists + obvious calls)

- **Single job, single keychain lifecycle.** Create a temp keychain at job start, delete it in an `if: always()` cleanup step at the end. Avoids cross-job artifact passing and never leaks the `.p12`/`.p8` even if a step fails mid-flight.
- **Identity is the SHA-1 hash detected at runtime** from the imported keychain. Exported to `$GITHUB_ENV`; used by `codesign --sign "$IDENTITY"`. If detection fails, the step fails loud before any sign attempt.
- **`--timestamp` on `codesign` is non-negotiable.** Notarytool rejects without it (with a generic error). Spec calls it out so the implementation doesn't silently drop it.
- **Notarize step parses JSON status** (`--output-format json` + `jq -r .status`). On non-`Accepted`, runs `xcrun notarytool log <id>` automatically before exiting non-zero. Debug never goes blind.
- **Staple runs on the `.app` in place.** The pre-notarization zip (`*-notary.zip`) becomes stale after staple writes the ticket inside the bundle; the **shipping zip is re-zipped** from the stapled `.app` (`DevTray-${VER}.zip`, same naming as v0.8).
- **EdDSA signing runs on the FINAL stapled zip.** Same `sign_update` invocation as v0.8 — just pointed at the post-staple re-zip. Sparkle on the user side verifies EdDSA (integrity) and Gatekeeper sees the stapled ticket (no internet needed). Both protections compose.
- **Cleanup secrets in `if: always()`** — `security delete-keychain "$KEYCHAIN_PATH" || true` + `rm -f /tmp/key.p8 "$RUNNER_TEMP/cert.p12"`. Critical: this MUST run on workflow failure too.
- **Release notes for the Sparkle compat note** — `release.yml` keeps `--generate-notes`. The compat banner is a **one-time author-only edit** to the v0.10.0 release body (via GitHub UI or `gh release edit`) after the workflow finishes. No `release-notes.sh` infra for a one-shot concern.
- **Version bump lives in TWO places** (`project.yml` `info.properties` + `App/DevTrayApp/Info.plist`) → `0.10.0` / `10`. `xcodegen generate` after the `project.yml` edit (`.xcodeproj` is gitignored).
- **CI stays on Xcode 16.2** (v0.8 carried lesson). The notarize/stapler tools come from `xcrun` and ship with Xcode — no install step needed.

## Components

### 1. `release.yml` — the 10-step insert + 1-step replace + 1-step append

**Trigger / unchanged steps** (no edits):
- `push` of tags matching `v*`; `permissions: contents: write`.
- `actions/checkout@v4` (full history).
- `sudo xcode-select -s /Applications/Xcode_16.2.app`.
- `brew install xcodegen`; `xcodegen generate`.
- **Read version** + tag-vs-plist guard.
- **Guard against placeholder pubkey** (`SUPublicEDKey`).
- **Build (Release, unsigned)** — `xcodebuild … CODE_SIGNING_ALLOWED=NO build`.
- **Download Sparkle tools** (`sparkle-project/Sparkle@2.9.2`).
- **EdDSA-sign the archive** — points at the final stapled zip path; no other changes.
- **Create GitHub Release** — `gh release create` with the final zip; `--generate-notes` stays.
- **Update appcast on `main`** — `scripts/appcast.sh` + commit + push (unchanged).

**Removed:**
- "Ad-hoc sign with hardened runtime" (the entire `codesign --sign -` step from v0.8).

**Inserted between "Build" and "Zip"** (10 new steps, in order):

1. **Setup signing env** — define once in `$GITHUB_ENV`:
   ```bash
   KEYCHAIN_PATH="$RUNNER_TEMP/build.keychain-db"
   APP_PATH="build/Build/Products/Release/DevTrayApp.app"
   ```
   Centralizes paths so subsequent steps don't re-derive them.

2. **Import Developer ID cert** (env: `DEVELOPER_ID_P12_BASE64`, `DEVELOPER_ID_P12_PASSWORD`):
   ```bash
   echo "$DEVELOPER_ID_P12_BASE64" | base64 --decode > "$RUNNER_TEMP/cert.p12"
   security create-keychain -p "" "$KEYCHAIN_PATH"
   security set-keychain-settings -lut 21600 "$KEYCHAIN_PATH"   # 6h, no auto-lock
   security unlock-keychain -p "" "$KEYCHAIN_PATH"
   security import "$RUNNER_TEMP/cert.p12" -P "$DEVELOPER_ID_P12_PASSWORD" \
     -A -t cert -f pkcs12 -k "$KEYCHAIN_PATH"
   security set-key-partition-list -S apple-tool:,apple: -s -k "" "$KEYCHAIN_PATH"
   security list-keychain -d user -s "$KEYCHAIN_PATH" $(security list-keychain -d user | tr -d '" ')
   ```
   The `set-key-partition-list` is the magic that prevents codesign from prompting for keychain access on each invocation in CI.

3. **Detect signing identity** — SHA-1 hash, exported to `$GITHUB_ENV`:
   ```bash
   IDENTITY=$(security find-identity -v -p codesigning "$KEYCHAIN_PATH" \
     | awk '/Developer ID Application/ {print $2; exit}')
   if [ -z "$IDENTITY" ]; then
     echo "::error::Developer ID Application identity not found in imported keychain"
     exit 1
   fi
   echo "IDENTITY=$IDENTITY" >> "$GITHUB_ENV"
   ```

4. **Sign with Developer ID** (hardened runtime + secure timestamp):
   ```bash
   codesign --force --deep --options runtime --timestamp \
     --sign "$IDENTITY" \
     --keychain "$KEYCHAIN_PATH" \
     "$APP_PATH"
   ```
   The `--keychain` flag scopes the lookup to the temp keychain (avoids picking up an unrelated identity if the runner ever caches state).

5. **Verify signature** — fail loud if codesign produced something Gatekeeper won't honor:
   ```bash
   codesign --verify --deep --strict --verbose=2 "$APP_PATH"
   codesign -dvv "$APP_PATH" 2>&1 | grep -E 'TeamIdentifier|Authority' || true
   ```
   The `grep` is informational (logs the team identifier for debug), never the failure gate.

6. **Decode ASC API key** (env: `APPSTORE_CONNECT_API_KEY_P8`):
   ```bash
   echo "$APPSTORE_CONNECT_API_KEY_P8" | base64 --decode > /tmp/key.p8
   chmod 600 /tmp/key.p8
   ```

7. **Zip for notarytool**:
   ```bash
   ditto -c -k --keepParent "$APP_PATH" "DevTray-${{ steps.ver.outputs.short }}-notary.zip"
   ```

8. **Notarize (`--wait` + JSON parse + log-on-failure)** (env: `APPSTORE_CONNECT_KEY_ID`, `APPSTORE_CONNECT_ISSUER_ID`):
   ```bash
   set -e
   SUBMISSION_JSON=$(xcrun notarytool submit \
     "DevTray-${{ steps.ver.outputs.short }}-notary.zip" \
     --key /tmp/key.p8 \
     --key-id "$APPSTORE_CONNECT_KEY_ID" \
     --issuer "$APPSTORE_CONNECT_ISSUER_ID" \
     --wait \
     --output-format json)
   echo "$SUBMISSION_JSON"
   STATUS=$(echo "$SUBMISSION_JSON" | jq -r '.status')
   SUBMISSION_ID=$(echo "$SUBMISSION_JSON" | jq -r '.id')
   if [ "$STATUS" != "Accepted" ]; then
     echo "::error::Notarization failed: status=$STATUS, submission=$SUBMISSION_ID"
     xcrun notarytool log "$SUBMISSION_ID" \
       --key /tmp/key.p8 \
       --key-id "$APPSTORE_CONNECT_KEY_ID" \
       --issuer "$APPSTORE_CONNECT_ISSUER_ID"
     exit 1
   fi
   ```
   Expect 5–15 min typical wait; first submission per account may go longer (not a bug).

9. **Staple ticket**:
   ```bash
   xcrun stapler staple "$APP_PATH"
   xcrun stapler validate "$APP_PATH"
   ```
   The `validate` is a cheap post-condition that catches a corrupted staple before we re-zip.

10. **Re-zip stapled app (final shipping artifact)**:
    ```bash
    ditto -c -k --keepParent "$APP_PATH" "DevTray-${{ steps.ver.outputs.short }}.zip"
    ```
    The `id: zip` output (used by EdDSA-sign + release create + appcast) points at this final zip — drop-in replacement for the v0.8 zip step.

**Appended (after appcast step):**

**Cleanup secrets** — `if: always()`:
```bash
security delete-keychain "$KEYCHAIN_PATH" || true
rm -f /tmp/key.p8 "$RUNNER_TEMP/cert.p12"
```
Must NOT leak secrets even if any prior step fails.

### 3. Sparkle compat release-notes banner (author-only, one-shot)

After the workflow publishes v0.10.0, the author edits the GitHub release body (UI or `gh release edit v0.10.0 --notes-file …`) to PREPEND:

```markdown
> ⚠️ **One-time manual update for v0.8.x/v0.9.x users**
> Earlier betas were ad-hoc signed; this release is signed with a real Developer ID and notarized by Apple. Sparkle's code-signing consistency check blocks the team-identifier change, so in-app auto-update from v0.8.x/v0.9.x → v0.10.0 will refuse. **Download this release manually and replace the app once.** Future updates (v0.10 → v0.11 → …) auto-install normally.
```

Subsequent releases (v0.10.1+, v0.11.0+) need no special note. This is a one-time concern; no `release-notes.sh` infrastructure justified.

## Manual (author-only) vs automatable split

This is the T10-style split that v0.8 established — agent lists the manual steps, author runs them between agent tasks.

**Author-only (machine / Apple account):**
1. Create **Developer ID Application** cert (Xcode → Settings → Accounts → Manage Certificates → `+` → Developer ID Application). Lands in login Keychain.
2. Export from Keychain Access → File → Export Items → `.p12` with a password.
3. `base64 -i cert.p12 -o cert.p12.b64`.
4. ASC → Users and Access → Integrations → App Store Connect API → create key with role **Developer** → download `.p8` (UMA vez só) + note **Key ID** + **Issuer ID**.
5. `base64 -i AuthKey_*.p8 -o key.p8.b64`.
6. Add **5 new GitHub secrets** (`SPARKLE_PRIVATE_KEY` from v0.8 stays untouched):
   - `DEVELOPER_ID_P12_BASE64` — content of `cert.p12.b64`
   - `DEVELOPER_ID_P12_PASSWORD` — the export password
   - `APPSTORE_CONNECT_API_KEY_P8` — content of `key.p8.b64`
   - `APPSTORE_CONNECT_KEY_ID` — ~10-char string
   - `APPSTORE_CONNECT_ISSUER_ID` — UUID
   Using `gh secret set X < file` after `gh auth switch --user AlexandreAkao` (dual-account ritual).
7. After PR merges to `main`, push annotated tag `v0.10.0` (this fires the workflow).
8. Run the manual smoke (see Testing) on a quarantine-tagged zip.
9. Edit the v0.10.0 release body on GitHub to PREPEND the Sparkle compat banner (one-shot).

**Agent / automatable / locally verifiable:**
- Edit `release.yml` (10 inserted steps + 1 replaced + 1 appended cleanup).
- Bump version in both files; `xcodegen generate`.
- YAML structural lint (`actionlint` if installed; otherwise rely on the workflow parse step on first dry-run tag).
- `xcodebuild -project DevTray.xcodeproj -scheme DevTrayApp build` — confirm local build still produces `DevTrayApp.app` (no behavior change expected; only version metadata).
- `./scripts/test.sh` — full suite stays green.
- `swiftformat --lint .` — should be no-op (no Swift edits).

## Carried lessons (reinforced from memory)

- **CI uses Xcode 16.2**, not 15.4. `sudo xcode-select -s /Applications/Xcode_16.2.app`. The `xcrun notarytool` / `xcrun stapler` binaries ship with Xcode — no install step.
- **Scheme is `DevTrayApp`, NOT `DevTray`.** `xcodebuild -scheme DevTray …` fails with a misleading "unable to resolve module dependency: DevTrayCore" cascade.
- **App bundle is `DevTrayApp.app`** (target name; the displayed "DevTray" in the UI is `CFBundleDisplayName`). v0.8 path lesson.
- **Version bump in TWO places** (`project.yml` `info.properties` + `App/DevTrayApp/Info.plist`).
- **`xcodegen generate` after `project.yml` edits**; `.xcodeproj` is gitignored. **NEVER hand-edit `project.pbxproj`.**
- **`gh` dual-account:** repo is `AlexandreAkao` (SSH push always works); active `gh` user is usually `AlexandreAkiraAkao` — `gh auth switch --user AlexandreAkao` before `gh secret set` / `gh release edit` / any API op; restore after.
- **Release ritual (v0.8 carryover):** PR merge ff to `main` + author creates annotated `v0.10.0` tag → tag triggers `release.yml`. Local branch deleted post-merge; remote ref kept.
- **`SPARKLE_PRIVATE_KEY` secret + `SUPublicEDKey` in Info.plist (`ZxyqDIzwmdH+/nfaxtGzTqG+ZZ3ZgNob8Ei0gS3k948=`) stay untouched.** DO NOT recreate the EdDSA key — that would break auto-update FOREVER for v0.8/v0.9 users.
- **`--timestamp` on codesign is non-negotiable for notarization.** Apple rejects without it (generic error).
- **`notarytool log <submission-id>`** is how you debug a `--wait` failure. Always pair the submit step with an automatic log-fetch on non-`Accepted` status.
- **Sparkle 2 code-signing consistency check** (brainstorm #5) — v0.8/v0.9 testers must redownload v0.10.0 manually ONE time. Release-body banner covers it.
- **Notarytool first submission per account is slow**; subsequent ones speed up. Don't read 20-min waits on the first try as a workflow bug.
- **Tag fires the release CI** (v0.8 ritual). **For this cycle, DO NOT tag until all 5 new secrets are set AND a sanity check has confirmed cert import works** — the placeholder-pubkey guard does NOT cover missing Developer ID secrets. The workflow will fail loud at the import step if a secret is missing, but burning a half-tagged release is messy. (Sanity: `gh secret list | grep DEVELOPER_ID && gh secret list | grep APPSTORE_CONNECT` should show all 5.)
- **SwiftFormat lint is active on CI** (v0.9). Run `./scripts/format.sh` before committing — though YAML doesn't pass through SwiftFormat, so for this cycle's edits it's a no-op.
