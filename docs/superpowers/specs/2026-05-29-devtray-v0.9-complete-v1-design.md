## Goal

The v0.x cycles delivered all 13 tools + persistence + Spotlight + release infra, but several **v1-spec features beyond the tools were never built**. This cycle closes the ones worth doing (the rest are explicitly descoped below):

1. **JWT verify + encode (HS256 + RS256)** — today `JWTToolKit` is decode-only.
2. **File input for Hash and Base64** — today both are text-only.
3. **Sidebar grouped by category + `Cmd+1…9`** — today the sidebar is a flat search list.
4. **Preferences "Tools" tab** — enable/disable individual tools (persisted).
5. **SwiftFormat lint in CI** — not present today.
6. **Complete the XCUITest smoke** — today only the launch scenario exists.

**Notarization / Developer ID** (the original "v0.9" label in the roadmap) **moves to a later cycle** — it depends on the paid Apple Developer account and is independent of this work.

## Architecture / packaging decisions (follows what exists)

- **No new external dependency.** JWT RS256 uses **Security framework** (`SecKey*`), HS256 uses **CryptoKit** (`HMAC<SHA256>`) — both system frameworks, linked implicitly. File input uses **AppKit `NSOpenPanel`** (app is unsandboxed; no entitlement needed, same as v0.6 snippet import).
- **Tool capability changes stay inside each tool package** (`JWTToolKit`, `HashToolKit`/view, `Base64Tool…`). Sidebar/Tools-tab changes touch `DevTrayUI` (`PopoverRoot`) + the app target + a new small `ToolPreferences` type in **DevTrayCore** (mirroring the existing `UsageStore`/env-key patterns).
- **Version bump in TWO places** (`project.yml` + `App/DevTrayApp/Info.plist`) → `0.9.0` / `9`. Run `xcodegen generate` after any `project.yml` change (none expected unless a new file needs a new target — these are package-internal or app-internal files auto-picked-up; **a new file in `App/DevTrayApp/` is auto-included by `xcodegen generate`, do NOT hand-edit `project.pbxproj`** — v0.8 lesson).
- **CI is on Xcode 16.2 / Swift 6 SDK** (v0.8 lesson — not 15.4). New code must compile there; SwiftUI helpers reading `@MainActor` state are fine under the Swift 6 SDK, but keep `@MainActor` explicit on non-`body` helpers (v0.8 SnippetsTool lesson).

## Components

### 1. JWT — verify + encode (HS256 + RS256)

**`JWTToolKit` engine additions** (`JWTEngine` gains, decode stays):
```swift
public enum JWTAlgorithm: String { case hs256 = "HS256", rs256 = "RS256" }

public enum JWTEngine {
    // existing: decode(_:) -> Result<DecodedJWT, ToolError>
    public static func verify(token: String, algorithm: JWTAlgorithm, key: String) -> Result<Bool, ToolError>
    public static func encode(headerJSON: String, claimsJSON: String, algorithm: JWTAlgorithm, key: String) -> Result<String, ToolError>
}
```
- **HS256** (`key` = shared secret, UTF-8): `HMAC<SHA256>.authenticationCode(for: signingInput, using: SymmetricKey(data: secret))`; base64url-encode for `encode`, constant-time compare for `verify`. `signingInput = base64url(header) + "." + base64url(payload)`.
- **RS256** (`key` = PEM): parse PEM → strip `-----BEGIN/END …-----` + whitespace → base64-decode to DER → `SecKeyCreateWithData(der, [kSecAttrKeyType: kSecAttrKeyTypeRSA, kSecAttrKeyClass: .public/.private])`. Verify: `SecKeyVerifySignature(_, .rsaSignatureMessagePKCS1v15SHA256, signingInput, sig, &err)`. Encode: `SecKeyCreateSignature(_, .rsaSignatureMessagePKCS1v15SHA256, signingInput, &err)`.
  - **PEM-format caveat (documented limitation):** `SecKeyCreateWithData` expects **PKCS#1** RSA DER (`RSAPublicKey`/`RSAPrivateKey`). PKCS#8 / SPKI (`BEGIN PUBLIC KEY` / `BEGIN PRIVATE KEY`) wrap the key in an algorithm header that must be stripped first. The engine handles PKCS#1 (`BEGIN RSA …KEY`) directly and attempts the common SPKI/PKCS#8 unwrap; anything it can't parse → `ToolError.invalidInput(reason:)` naming the problem. Unsupported exotic encodings are an explicit limitation, not a silent failure.
- All failures (bad key, malformed token, alg mismatch) → `ToolError.invalidInput`/`.parseFailure` with a clear reason.

**`JWTToolView`:** keep the live decode (header/payload/signature). Add:
- an **algorithm `Picker`** (HS256 / RS256),
- a **key field** — single-line secret for HS256, multi-line `CodeEditor` for the PEM when RS256,
- a **Verify** affordance showing ✓ valid / ✗ invalid (or the error),
- an **Encode** section: header JSON + claims JSON editors → "Generate token" → output token (copyable). For RS256 encode the key field is the **private** PEM; a small note flags that the private key never leaves the app.

### 2. Hash & Base64 — file input

Both tools gain a **"Choose file…"** button in their views:
- `NSOpenPanel` (allowsMultipleSelection false) → `Data(contentsOf: url)` → feed the **existing engine** over the bytes (engines already operate on `Data`/bytes; only the input source is new).
- **Hash:** hashes the file bytes; shows the file name + each digest, same output UI as text mode.
- **Base64:** encodes the bytes. Output can be large → show **byte count + a Copy button** rather than dumping megabytes into the `TextEditor` (guard a display cap, e.g. only render the first N chars with a "copy full output" action).
- Text mode stays the default; file mode is an alternate input. Read errors → `ToolError.storageFailure`/`.invalidInput`.

### 3. Sidebar grouped by category + `Cmd+1…9`

In `DevTrayUI/PopoverRoot.swift`:
- When the search field is **empty**, render the sidebar **grouped by `ToolCategory`** — iterate `ToolCategory.allCases` (the enum's case order = section order), a section header per non-empty category (`category.displayName`), tools listed under it. When the search field has **text**, fall back to the current **flat filtered list** (no headers).
- **`Cmd+1…9`** selects the N-th tool in the current visible order (`.keyboardShortcut("1", modifiers: .command)` … through 9), bound on the popover. Maps to the flattened visible-tool order so it works in both grouped and filtered modes.
- Respects the enabled/disabled set from §4 (disabled tools don't appear).

### 4. Preferences "Tools" tab (enable/disable)

- **`ToolPreferences`** — a small `ObservableObject` (or `@Observable`) backed by **`UserDefaults`**, holding the set of **disabled** tool IDs (default: empty = all enabled). API: `isEnabled(_ id: ToolID) -> Bool`, `setEnabled(_:_:)`, `enabledTools(from:) -> [AnyTool]`. Lives in DevTrayCore (next to the usage/snippet env-key patterns) with an `@Environment` key, injected like the other stores.
- **Consumers filter by it:** `PopoverRoot` sidebar (§3) and the **Spotlight ranker** both exclude disabled tools — consistency (a disabled tool is gone from both surfaces). (The registry itself stays the full set; filtering happens at the consumer, mirroring how `usageStore` is consumed.)
- **Tools tab** (`App/DevTrayApp/DevTrayApp.swift`, a new `ToolsTab` in the `SettingsView` `TabView`): lists every registered tool (icon + name + category) with a `Toggle`. Disabling all → the popover/Spotlight show an empty state ("All tools are disabled — enable some in Settings → Tools").

### 5. SwiftFormat lint in CI

- Add a committed **`.swiftformat`** config (default ruleset + the repo's indentation/line conventions — derived from the existing code so it's a no-op on current files).
- Add a **lint step** to `.github/workflows/test.yml`: install SwiftFormat and run `swiftformat --lint .` (fails on unformatted code; **does not** auto-format in CI). Runs on **`ubuntu-latest`** (SwiftFormat runs on Linux) — fold into the existing `discover` job or a small dedicated `lint` job, to avoid spending macOS minutes.
- A `scripts/format.sh` (or documented `swiftformat .`) for local auto-fix before pushing.

### 6. Complete the XCUITest smoke

`Tests/DevTrayAppUITests/SmokeTests.swift` currently asserts launch. Add, **reliable-first**:
- **Popover opens:** best-effort click the `MenuBarExtra` status item (via the system menu-bar element) and assert the popover content appears. If this proves flaky on CI, **demote to a controller/integration-level assertion** rather than chase a flaky UI click.
- **Spotlight opens:** instead of simulating a system-global hotkey (unreliable in CI), exercise **`SpotlightController.toggle()`** directly (integration test in the app target) and assert the panel becomes visible. The real global-hotkey path stays covered by the manual pre-release checklist.
- Honest principle: **a reliable integration test beats a flaky literal XCUITest.** The plan picks the most reliable form per scenario.

## Testing

- **JWT (`JWTToolKitTests`):** HS256 verify (valid/invalid secret), HS256 encode→decode round-trip; RS256 verify against a known PKCS#1 PEM public key + known token; RS256 encode→verify round-trip with a test key pair; malformed PEM → `invalidInput`; algorithm mismatch → error. (Test key pairs as fixtures.)
- **Hash/Base64 file input:** engines already unit-tested on bytes; add a test feeding a temp file's `Data` through the engine path; large-output base64 display-cap logic unit-tested.
- **`ToolPreferences`:** enabled/disabled persistence round-trip (UserDefaults suite for tests), default-all-enabled, filter helper.
- **Sidebar grouping:** a `ToolRegistry`/grouping helper test that tools bucket into the right `ToolCategory` order; disabled tools excluded.
- **Full suite:** `./scripts/test.sh`; **`swiftformat --lint .`** must pass (run it before pushing).
- **App build:** `xcodebuild -project DevTray.xcodeproj -scheme DevTrayApp build` (scheme is **`DevTrayApp`**). CI on **Xcode 16.2**.
- **Manual smoke:** Clean Build Folder first; JWT verify/encode for both algorithms; hash/base64 a file; sidebar shows category sections + `Cmd+1…9` jumps; disable a tool in Settings → it vanishes from sidebar AND Spotlight; re-enable.
