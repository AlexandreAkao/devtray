# DevTray v0.7 — Close the v1 tool surface (regex · diff · color · cron · yaml) · Design

**Date:** 2026-05-27
**Cycle:** v0.7 (adds the final 5 tools; completes the v1 `Feature set` of 13 tools)
**Branch:** `feat/v0.7-tools` (in place, no worktree — manual smoke tests run better on the main checkout, same as v0.5/v0.6)
**Predecessor:** v0.6 spec `docs/superpowers/specs/2026-05-25-devtray-v0.6-export-settings-design.md`, plan `docs/superpowers/plans/2026-05-25-devtray-v0.6-export-settings.md`
**Reference:** general design `docs/superpowers/specs/2026-05-20-devtray-design.md` (Feature set v1 table, lines 53–57)

## Goal

Take DevTray from **8 → 13 tools**, completing the v1 feature set. Five new tools, each an independent `XxxToolKit` (pure engine + optional clipboard matcher) + `XxxTool` (SwiftUI view + `Tool` conformance) slice following the consolidated 8-tool pattern. Per the v1 table:

1. **`regex`** — Regex tester: live match, capture groups, replace.
2. **`diff`** — Text diff: side-by-side or unified.
3. **`color`** — Color converter: HEX ↔ RGB ↔ HSL + picker.
4. **`cron`** — Cron expression: parse + human description + next executions.
5. **`yaml`** — YAML ↔ JSON, bidirectional.

## Brainstorm decisions

| Question | Decision | Why |
|---|---|---|
| Batch size | **All 5 in one cycle** | Each tool is an independent Kit+Tool slice — ideal for isolated subagent task blocks. Reaches the full v1 surface (13 tools) in one version rather than spreading across v0.7+v0.8. |
| External dep vs hand-rolled | **Yams for `yaml` only; other 4 hand-rolled** on Foundation/stdlib | Hand-rolling YAML is infeasible; the v1 design names Yams "the only viable Swift option." regex/diff/color/cron have bounded, well-understood hand-rolled solutions with no dep cost. |
| Regex engine | **`NSRegularExpression`** | Battle-tested; capture groups via `NSRange` are easy; replace templates (`$1`) are built in; simple synchronous API matching the existing sync-engine pattern. Swift `Regex` runtime-pattern API is fiddlier for user-supplied patterns. |
| Diff algorithm / granularity / view | **Line-level, hand-rolled Myers, side-by-side primary + unified toggle** | Myers SES gives control over hunks so one engine drives both side-by-side AND unified-with-context. `CollectionDifference` is weaker for contiguous hunks. Char/word-level is a possible later refinement, out of scope now. |
| Clipboard matchers | **`color → strong`, `cron → weak`; none for regex/diff/yaml** | Hex/rgb/hsl is a high-confidence, low-collision color signal. 5 valid cron fields is a real but text-colliding signal → weak. yaml "looks like YAML" collides with the `json` tool and plain `key: value` text → no matcher. regex/diff have no detectable clipboard signal. |
| Tool naming | **`YAMLTool`/`YAMLToolKit`** (uppercase acronym) | Matches existing `JSONTool`/`JWTTool`/`UUIDTool` convention. (`id` is still `"yaml"`.) The other four use `RegexTool`, `DiffTool`, `ColorTool`, `CronTool`. |
| `ToolCategory` mapping | regex→`.text`, diff→`.text`, color→`.formatting`, cron→`.time`, yaml→`.formatting` | No `color` category exists; `.formatting` fits a value converter. yaml→`.formatting` like `json`. cron→`.time` like Timestamp. User-confirmed. |

## Packaging (architecture decision, not a brainstorm question)

Each tool is a Swift package at `Packages/Tools/XxxTool/` with two products, modeled on `Packages/Tools/URLTool` (the leanest existing tool):

- **`XxxToolKit`** — pure engine, depends **only on DevTrayCore** (not DevTrayUI, not DevTrayStorage). Contains `XxxEngine` returning `Result<_, ToolError>` + `XxxClipboardMatcher` (only for color/cron).
- **`XxxTool`** — SwiftUI view, depends on `XxxToolKit` + DevTrayCore + DevTrayUI. Contains `enum XxxTool: Tool` (the conformance: `id`, `displayName`, `iconName`, `keywords`, `category`, `makeView()`, plus the one-line `clipboardMatch` extension calling the matcher) + `XxxToolView`.
- **`XxxToolKitTests`** test target.

`Package.swift` mirrors `URLTool/Package.swift` exactly (two products, two targets, one testTarget) — except `YAMLTool` adds Yams as a dependency on `YAMLToolKit` (see §6).

**SwiftPM empty-target gotcha (carried since v0.2):** the engine task runs before the view task creates `Sources/XxxTool/`, and SwiftPM rejects a target referenced in a product with no sources ("target referenced in product is empty"). Bake a 2-line stub into `Sources/XxxTool/XxxTool.swift` at package-creation time (overwritten later by the real conformance) — the v0.5 approach, which worked.

## Components

The five engines are independent; each is specified below with its public surface, view, and matcher. All engines are **synchronous and pure** — no async, so tests need no `let`-before-assert workaround (that rule only bites `try await` asserts).

### 1. `regex` — RegexTool (category `.text`)

`RegexEngine` over `NSRegularExpression`:

```swift
public struct RegexFlags: OptionSet { caseInsensitive, multiline, dotMatchesLineSeparators }
public struct RegexMatch { let range: Range<String.Index>; let value: String; let groups: [RegexGroup] }
public struct RegexGroup { let index: Int; let name: String?; let range: Range<String.Index>?; let value: String? }

public enum RegexEngine {
    public static func evaluate(pattern: String, flags: RegexFlags, input: String) -> Result<[RegexMatch], ToolError>
    public static func replace(pattern: String, flags: RegexFlags, input: String, template: String) -> Result<String, ToolError>
}
```

- Compile failure → `.parseFailure(reason: <NSError localizedDescription>, hint: nil)`. Empty pattern → empty result, not an error.
- Capture groups: walk `NSTextCheckingResult.numberOfRanges`; a group that didn't participate has `range == nil`. Named groups recovered by scanning the pattern for `(?<name>…)` and mapping to index order.
- `replace` uses `NSRegularExpression.stringByReplacingMatches` with the user's template (`$1`, `$0`).
- **View (`RegexToolView`):** pattern `TextField`, flag toggles, multiline test-input `TextEditor`, live results (recompute on change of pattern/flags/input), capture-group list per match, replace section (template field + output). No matcher.

### 2. `diff` — DiffTool (category `.text`)

`DiffEngine`: hand-rolled Myers shortest-edit-script over `[Substring]` lines.

```swift
public enum DiffRowKind { case equal, insert, delete }
public struct DiffRow { let kind: DiffRowKind; let text: String; let leftLine: Int?; let rightLine: Int? }
public struct DiffHunk { let header: String; let rows: [DiffRow] }   // unified, with context

public enum DiffEngine {
    public static func diffLines(_ a: String, _ b: String) -> [DiffRow]        // side-by-side source of truth
    public static func unifiedHunks(_ a: String, _ b: String, context: Int = 3) -> [DiffHunk]
}
```

- `diffLines` produces the full row sequence (line numbers assigned per side). `unifiedHunks` groups runs of changes plus `context` surrounding equal lines into `@@`-style hunks; all-equal input → no hunks.
- **View (`DiffToolView`):** two input `TextEditor`s (left/right), a `Picker` view mode (Side-by-side | Unified), rendered diff below (color-coded insert/delete). No matcher.

### 3. `color` — ColorTool (category `.formatting`)

`ColorEngine`: pure conversions; `ColorValue` holds canonical RGBA (r/g/b 0–255, a 0–1).

```swift
public struct ColorValue: Equatable { let r, g, b: Int; let a: Double
    var hex: String           // #RRGGBB or #RRGGBBAA when a < 1
    var rgbString: String     // rgb(…) / rgba(…)
    var hslString: String     // hsl(…) / hsla(…)
}
public enum ColorEngine {
    public static func parse(_ input: String) -> Result<ColorValue, ToolError>   // #RGB/#RRGGBB/#RRGGBBAA, rgb()/rgba(), hsl()/hsla()
}
```

- Parse failure → `.invalidInput(reason:)`. HEX↔RGB↔HSL math hand-rolled and unit-tested at known anchor colors (black/white/primary hues + a round-trip property check).
- `ColorClipboardMatcher` → **strong** when trimmed clipboard matches a hex/rgb/hsl shape (reuses the same recognizers as `parse`, returns `ClipboardMatchScore(.strong)`).
- **View (`ColorToolView`):** text input; three copyable outputs (HEX / RGB / HSL); SwiftUI `ColorPicker` bound two-way; swatch. Editing any field reconciles the rest. Reads `@Environment(\.preloadBus) var preloadBus: PreloadBus` (explicit type annotation) + `applyPendingPreloadIfMatches()` for smart-paste, exactly like `URLToolView`.

### 4. `cron` — CronTool (category `.time`)

`CronEngine`: parse a 5-field cron expression (`minute hour day-of-month month day-of-week`).

```swift
public struct CronExpression: Equatable { /* per-field allowed value sets */ }
public enum CronEngine {
    public static func parse(_ expr: String) -> Result<CronExpression, ToolError>
    public static func humanDescription(_ e: CronExpression) -> String
    public static func nextExecutions(_ e: CronExpression, from: Date, count: Int = 5, timeZone: TimeZone = .current) -> [Date]
}
```

- Field syntax: `*`, `a`, `a-b`, `a,b,c`, `*/n`, `a-b/n`; month names (`JAN`…`DEC`) and day names (`SUN`…`SAT`); macros `@hourly @daily @midnight @weekly @monthly @yearly @annually`. Out-of-range/garbage → `.parseFailure(reason:hint:)` naming the bad field.
- `humanDescription`: e.g. `"At 09:00, Monday through Friday"`. Covers the common shapes; falls back to a literal field-by-field description for exotic expressions.
- `nextExecutions`: minute-stepping from `from`, collecting matches in `timeZone`, capped (e.g. 5 years) to avoid infinite loops on impossible combos.
- `CronClipboardMatcher` → **weak** when the clipboard is exactly 5 whitespace-separated tokens that all validate as cron fields (macros do **not** match — too text-colliding).
- **View (`CronToolView`):** expression `TextField`, human description, next-5 executions list (local tz). Reads `@Environment(\.preloadBus): PreloadBus` + `applyPendingPreloadIfMatches()` for smart-paste.

### 5. `yaml` — YAMLTool (category `.formatting`)

`YAMLEngine` over **Yams**:

```swift
public enum YAMLEngine {
    public static func yamlToJSON(_ yaml: String) -> Result<String, ToolError>   // pretty JSON
    public static func jsonToYAML(_ json: String) -> Result<String, ToolError>
}
```

- `yamlToJSON`: `Yams.load` → re-serialize via `JSONSerialization` (`.prettyPrinted` + `.sortedKeys`). `jsonToYAML`: `JSONSerialization.jsonObject` → `Yams.dump`. Parse errors (either side) → `.parseFailure(reason: <error>, hint: nil)`.
- **View (`YAMLToolView`):** input `TextEditor`, direction `Picker` (YAML→JSON / JSON→YAML) with a swap action, output. **No matcher** (avoids colliding with the `json` tool).

## Registration (4 places per new tool)

1. `App/DevTrayApp/DevTrayApp.swift` → `import XxxTool` + `r.register(XxxTool.self)` in `makeRegistry()`.
2. `project.yml` → `packages:` block (`XxxTool: path: Packages/Tools/XxxTool`) **and** `targets.DevTrayApp.dependencies:` (`- package: XxxTool`).
3. `scripts/test.sh` → add the package to the `PACKAGES` array (else its tests don't run).
4. Run **`xcodegen generate`** after editing `project.yml` (the `.xcodeproj` is gitignored and generated).

## Testing

- **Per-tool Kit unit tests (TDD):**
  - `RegexEngineTests`: match positions, multiple matches, capture groups (numbered + named + non-participating), each flag, replace templates, invalid pattern → `parseFailure`, empty pattern.
  - `DiffEngineTests`: identical input (all equal), pure insert, pure delete, mixed; unified hunks group with correct context; line numbering per side.
  - `ColorEngineTests` + `ColorClipboardMatcherTests`: parse each input format; HEX↔RGB↔HSL anchors + round-trip; invalid input → `invalidInput`; matcher strong on hex/rgb/hsl, nil on noise.
  - `CronEngineTests` + `CronClipboardMatcherTests`: each field syntax, names, macros; human descriptions for common shapes; nextExecutions correctness against known dates + tz; bad field → `parseFailure`; matcher weak on 5 valid fields, nil otherwise (incl. macros).
  - `YAMLEngineTests`: yaml→json and json→yaml round-trips on scalars/sequences/maps/nested; malformed input each way → `parseFailure`.
- **Full suite:** `./scripts/test.sh` (after adding all 5 packages to `PACKAGES`).
- **App build:** `xcodebuild -project DevTray.xcodeproj -scheme DevTrayApp build` — **scheme is `DevTrayApp`, not `DevTray`** (using `-scheme DevTray` fails with a misleading "unable to resolve module dependency: DevTrayCore" cascade).
- **Manual smoke:** Clean Build Folder (`⌥⇧⌘K`) first; verify all 5 tools open from the menu and Spotlight; verify color/cron smart-paste promotes in Spotlight (color strong, cron weak); verify regex/diff/yaml do **not** spuriously promote.

## Carried lessons relevant here

- **Scheme is `DevTrayApp`, not `DevTray`** (the v0.6 plan kept getting this wrong — don't repeat).
- **`clipboardMatch` must stay in the `Tool` protocol body** (it is) — otherwise generic callers use static dispatch and matchers become dead code (v0.4 latent bug). The concrete matcher lives in `XxxToolKit/XxxClipboardMatcher.swift`; the override is a one-line extension on `XxxTool`.
- **SwiftPM empty-target gotcha:** bake a 2-line stub in `Sources/XxxTool/` at package creation (overwritten by the view task).
- **`@Environment(\.xxx)` cross-module always needs an explicit type annotation** (`: PreloadBus`) or cold SwiftPM/SourceKit fails key-path inference.
- **`@Environment(\.preloadBus)` already injected** into both `PopoverRoot` and the Spotlight `NSHostingView` (v0.6) — color/cron views get preload in both surfaces for free by just reading the env. No new env seam is introduced this cycle, so no injection changes needed (v0.6 lesson: every Scene consuming an env store must inject it).
- **Version bump lives in two places** (`project.yml` + `Info.plist`).
- **Expect `Package.resolved` churn** from Yams; commit it (v0.4 KeyboardShortcuts precedent).
- **SourceKit cross-module lag is normal noise** — trust `swift test` (per package) + `./scripts/test.sh` + `xcodebuild -scheme DevTrayApp`; Clean Build Folder before the first manual smoke.
- **`gh` dual account:** repo is `AlexandreAkao` (SSH push works); active `gh` is usually `AlexandreAkiraAkao` — `gh auth switch --user AlexandreAkao` before any API op (PR), restore after.
- **Release pattern:** close the version with a PR merge to `main` + an annotated tag `v0.7.0` pushed; delete the local branch post-merge, keep the remote ref.

## Out of scope

- Char/word-level (intra-line) diff highlighting — possible later refinement; v0.7 is line-level.
- yaml clipboard matcher (collides with `json`); regex/diff matchers (no signal).
- 6-field cron (with seconds) and Quartz-specific syntax — 5-field POSIX-style only.
- Regex "explain this pattern" / cheat-sheet UI; diff 3-way merge; color palettes/named colors beyond parse — all beyond the v1 table specs.
- Sparkle auto-update + release CI — the next milestone toward v1.0, not this cycle.
