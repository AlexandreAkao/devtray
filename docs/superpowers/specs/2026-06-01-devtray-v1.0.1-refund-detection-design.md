# DevTray v1.0.1 — Reliable refund detection · Design

**Date:** 2026-06-01
**Cycle:** v1.0.1 (patch on top of v1.0.0 — first paying-user-ready hotfix). Ships as `1.0.1` / build `13`.
**Branch:** `feat/v1.0.1-refund-detection` (in place, no worktree — backend deploy runs better on the main checkout).
**Predecessor:** v1.0 spec `docs/superpowers/specs/2026-05-31-devtray-v1.0-paddle-launch-design.md`.
**Trigger:** v1.0 smoke caught that a real Paddle refund did not invalidate the minted license. Manual KV revoke was used as workaround; the gap blocks scaling to actual paying users.

## Goal

v1.0 ships a working mint flow. The refund flow has a load-bearing gap: Paddle dispatched no `adjustment.created` event our handler subscribed to recognized, and even if it had, the handler would have revoked at the wrong status. Two independent issues stacking.

**Issue 1 — Subscription + status-awareness mismatch.** Per Paddle Billing v1 docs (verified 2026-06-01 from `developer.paddle.com/webhooks/adjustments/adjustment-created` and `adjustment-updated`):

> *"Most refunds for live accounts require Paddle approval and are created as `pending_approval`. `adjustment.updated` occurs when the status changes to `approved` or `rejected`."*

The current v1.0 webhook handler:
- Subscribes only to `adjustment.created`
- Revokes immediately when `action === "refund"` regardless of `status`

For a live-account seller-initiated refund, the actual sequence is:
1. `adjustment.created` fires with `status: pending_approval` — **money has not moved yet**; revoking now is wrong
2. Paddle approves (no SLA documented; can be minutes to days)
3. `adjustment.updated` fires with `status: approved` — money is refunded; this is when revoke should happen
4. OR Paddle rejects → `adjustment.updated` fires with `status: rejected` — no money moves; license stays valid

We need to subscribe to `adjustment.updated` and gate revoke on `status === "approved"`.

**Issue 2 — Webhooks are inherently unreliable.** Even with the right subscription + handler logic, webhooks can be lost (network, Paddle downtime, our Worker timeout). For money-related state, that's not acceptable as the sole signal. A polling reconciliation fallback queries Paddle's API on a schedule and brings KV in line with authoritative state.

This cycle adds:
1. Subscription to `adjustment.updated` (author-manual; no code)
2. Status-aware handler that revokes only when adjustment is `approved`
3. Cron-based polling reconciliation that runs every 30 minutes, queries Paddle for each active license's transaction status, and revokes any whose transaction has been refunded

Also bundled: cleanup of the zombie license `3f365dd7-286d-42cb-a1cc-7e17ceab2d21` (a retroactive mint from a previously-refunded transaction; left over from v1.0 smoke).

**Not in scope (defer):**
- Pre-mint transaction status check (defense-in-depth at mint time) → v1.0.2 or later
- Per-heartbeat lazy refund check → v1.0.2 or later if cron lag becomes user-visible
- Custom app icon, e2e test infra, hero GIF — v1.0.1 stays focused on the refund hotfix

## Brainstorm decisions

| Question | Decision | Why |
|---|---|---|
| **Subscribe to which additional events?** | `adjustment.updated` only | The only event that fires when a live-account refund actually completes approval. Other candidates (`transaction.updated`, `transaction.canceled`) are not specced as refund-related per Paddle docs. |
| **Handler status gate** | Revoke only when `status === "approved"`. Treat `pending_approval` and `rejected` as no-op + markEventProcessed. | Per docs: `pending_approval` means money hasn't moved; `rejected` means it won't. Revoking on either would either be premature or wrong. |
| **Reconciliation strategy** | Cron polling every 30 minutes, hits `GET /transactions/{id}` for each license whose KV record is `revoked: false`. | Simple, defensive, polls authoritative source. 30 min lag is acceptable for refund detection (no UX urgency). |
| **Polling frequency** | 30 minutes | Refunds aren't time-critical to invalidate from a fraud-prevention POV at v1.0.1 scale. Higher frequency (5-15 min) is unnecessary API cost. Lower (1h+) extends gap meaningfully. |
| **Polling scope** | All license records in `LICENSES` namespace where `revoked: false` AND age between 60s and 90 days. | Skip records younger than 60s (avoid mint race). Skip records older than 90d (past Paddle's refund window). |
| **Refund states to revoke on** | `refunded`, `partially_refunded` | Both mean money returned to buyer. For digital goods, partial refund = revoke (we can't deliver "half" a license). |
| **Reverse a revoke if Paddle rejects later?** | No — leave revoked records as revoked. | If Paddle rejected the refund AFTER we revoked (only happens via `adjustment.updated rejected` after we previously processed an `approved`), it means money was refunded then un-refunded. Rare; if it happens, handle manually via support email and KV edit. Auto-unrevoke adds complexity for negligible benefit. |
| **Cron via CF Workers `[triggers]`** | Yes | Built-in Cloudflare Workers Cron Triggers. Free tier supports it. No external scheduler needed. |
| **Rate-limit concern** | None at current scale | Paddle API limit is 30 req/sec. At v1.0.1 scale (tens of licenses), each sweep is ~50 GET calls in series — well under limit. Re-evaluate at 1000+ licenses. |
| **Zombie license `3f365dd7-...` cleanup** | One-off `wrangler kv key put` to flip `revoked: true`. Documented in author-manual steps. | One-shot leftover from v1.0 smoke; not worth code. |
| **App-side changes** | None | Existing heartbeat `/status` already reads `revoked: true` from KV and flips UI. The polling fix is server-only. |
| **Version bump** | `1.0.1` / build `13` | Patch release. Triggers Sparkle auto-update for v1.0 users (already shipped). |

## Architecture decisions (not brainstorm — follow from above)

- **Webhook subscription change is author-manual.** Spec lists the dashboard steps; agent can't change Paddle config.
- **Handler change is additive, not breaking.** New `case "adjustment.updated"` added; existing `adjustment.created` case modified to require `status === "approved"`. Existing tests need fixture updates (the `status` field becomes load-bearing).
- **Cron handler is a NEW entry point in the Worker.** CF Workers exposes a `scheduled(controller, env, ctx)` export alongside the existing default `fetch` handler. Both share `Env` bindings.
- **Reconciliation function is shared.** Same logic the cron calls is also exposed via a `POST /admin/reconcile` HTTP endpoint (gated by `ADMIN_TOKEN`) for manual triggering during smoke and ad-hoc audit.
- **Reconciliation never deletes data.** It only flips `revoked: false → true`. Idempotent: re-running on an already-revoked record is a no-op (the `revoked: false` filter excludes it).
- **No KV state for "last polled" timestamp.** At v1.0.1 scale, scanning all 100s of records on every sweep is cheap. If scale grows past ~5000 licenses, add cursor pagination + per-record `last_polled_at` to skip recently-checked records.
- **Sandbox vs prod isolation stays as-is.** Cron polls only `LICENSES` (prod namespace). `LICENSES_TEST` is sandbox; nobody is polling it. Acceptable for v1.0.1.

## Components

### 1. Backend webhook handler — `backend/license/src/routes/webhook.ts`

Existing `adjustment.created` case becomes:

```ts
case "adjustment.created":
case "adjustment.updated":
  if (event.data?.action !== "refund" && event.data?.action !== "chargeback") {
    await markEventProcessed(env, eventId, "skipped");
    return new Response("ok (ignored: non-refund adjustment)", { status: 200 });
  }
  if (event.data?.status !== "approved") {
    // pending_approval (waiting for Paddle to approve), rejected (won't process),
    // anything else — money has NOT moved, do not revoke.
    await markEventProcessed(env, eventId, "skipped");
    return new Response(`ok (status=${event.data?.status})`, { status: 200 });
  }
  return revokeByAdjustment(event, env, eventId);
```

Same case body handles both events. Tests updated to cover:
- `adjustment.created` with `status: pending_approval` → no revoke
- `adjustment.created` with `status: approved` → revoke (sandbox/auto-approve)
- `adjustment.updated` with `status: approved` → revoke (live flow)
- `adjustment.updated` with `status: rejected` → no revoke

### 2. Reconciliation library — `backend/license/src/lib/reconcile.ts` (new)

```ts
type ReconcileResult = {
  scanned: number;
  fetched: number;
  revoked: number;
  errors: number;
};

export async function reconcileRefunds(
  env: Env,
  fetchImpl: FetchImpl = fetch,
  now: number = Date.now(),
): Promise<ReconcileResult> {
  const result: ReconcileResult = { scanned: 0, fetched: 0, revoked: 0, errors: 0 };
  const apiBase = env.PADDLE_API_BASE_URL || "https://api.paddle.com";
  const list = await env.LICENSES.list();
  if (list.list_complete === false) {
    console.warn("[reconcile] KV list truncated — some licenses skipped this pass");
  }

  for (const key of list.keys) {
    result.scanned++;
    const rec = (await env.LICENSES.get(key.name, "json")) as LicenseRecord | null;
    if (!rec) continue;
    if (rec.revoked) continue;
    if (!rec.paddle_transaction_id) continue;

    const ageSec = now / 1000 - rec.created_at;
    if (ageSec < 60) continue;          // skip just-minted (avoid race with webhook)
    if (ageSec > 90 * 86400) continue;  // skip past Paddle's refund window

    result.fetched++;
    try {
      const res = await fetchImpl(`${apiBase}/transactions/${encodeURIComponent(rec.paddle_transaction_id)}`, {
        headers: { Authorization: `Bearer ${env.PADDLE_API_KEY}` },
      });
      if (!res.ok) {
        result.errors++;
        console.warn(`[reconcile] fetch failed license=${key.name} txn=${rec.paddle_transaction_id} status=${res.status}`);
        continue;
      }
      const body = (await res.json()) as { data?: { status?: string } };
      const txStatus = body?.data?.status;
      if (txStatus === "refunded" || txStatus === "partially_refunded") {
        rec.revoked = true;
        await env.LICENSES.put(key.name, JSON.stringify(rec));
        result.revoked++;
        console.log(`[reconcile] revoked license=${key.name} txn=${rec.paddle_transaction_id} status=${txStatus}`);
      }
    } catch (err) {
      result.errors++;
      console.error(`[reconcile] error license=${key.name} txn=${rec.paddle_transaction_id}`, err);
    }
  }
  return result;
}
```

### 3. Cron entry point — `backend/license/src/index.ts`

The existing default export is a Hono app (handles `fetch`). CF Workers supports an additional `scheduled` export on the same module:

```ts
import app from "./app"; // refactor: move Hono instance to its own file
import { reconcileRefunds } from "./lib/reconcile";

export default {
  fetch: app.fetch,
  scheduled: async (event: ScheduledController, env: Env, ctx: ExecutionContext) => {
    const result = await reconcileRefunds(env);
    console.log(`[cron] reconcile result scanned=${result.scanned} fetched=${result.fetched} revoked=${result.revoked} errors=${result.errors}`);
  },
};
```

(The current `src/index.ts` exports `app` directly as the default. Refactor it into a small wrapper that exports both handlers. Existing tests for routes don't change — they hit `handleX(req, env)` directly, not via `app.fetch`.)

### 4. Manual trigger route — `backend/license/src/routes/admin.ts` (extend)

Add a `POST /admin/reconcile` route alongside the existing `POST /admin/release-all`. Same `ADMIN_TOKEN`-gated pattern. Calls `reconcileRefunds(env)` and returns the result as JSON. Useful during smoke (force the sweep without waiting 30 min).

### 5. `wrangler.toml` — add cron trigger

```toml
[triggers]
crons = ["*/30 * * * *"]
```

(Every 30 min on the dot. CF Workers Cron Triggers are part of the free tier; no billing impact.)

### 6. App side — no changes

Existing heartbeat `/status` already returns `{ revoked: true }` when the KV record has `revoked: true`. The polling fix is server-only. App-side bump is only the version (1.0.1 / 13) for Sparkle to push the update.

## Risks & lessons (from v1.0 cycle)

- **Spec-time docs check is critical.** All 4 v1.0 cycle bugs (`transaction.refunded` non-existent, `customer.email` not embedded, magic link not a static URL, `data.origin` not a sandbox flag) AND this v1.0.1 bug (`pending_approval` status gate) would have been caught by reading current Paddle Billing v1 docs at spec time instead of relying on training-data knowledge. Carry to v1.0.2: for any external API integration, fetch the live spec docs FIRST during brainstorm, not after the bug is in production. Same lesson explicitly applies to future provider migrations.
- **Webhooks alone are not enough for money-related state.** Cron polling reconciliation is now part of the architecture and should stay. Any future money-event handling (subscriptions, plan changes, etc.) gets the same polling treatment from day 1.
- **CF Workers Cron Triggers register only on deploy.** First deploy after adding `[triggers]` config to `wrangler.toml` may not show the trigger active immediately — verify via `wrangler triggers list` or `wrangler deployments status` after deploy.
- **The /admin/reconcile endpoint is sensitive.** Gate it with `ADMIN_TOKEN` (already a secret). Logs the result but never the per-record details (no PII leak).
- **Zombie license cleanup is one-shot.** Don't generalize into a "clean orphans" feature — once cleaned, no recurrence (we fixed the underlying race in handler logic + polling catches future drift).
- **`wrangler.toml` `[triggers]` is a new config block** — confirm syntax compatible with the existing schema before committing (run `wrangler deploy --dry-run` to validate).
- **Notarytool stays warm from v1.0** — v1.0.1 release should be the fast (~2:25 min) path.
- **All v1.0 carryforward conventions apply:** Xcode 16.2, dual-account gh ritual, scripts/format.sh on Swift, version bump in both Info.plist + project.yml, etc.
