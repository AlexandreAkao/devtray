# DevTray v0.3 — Design Document

**Date:** 2026-05-21
**Status:** Design (awaiting plan)
**Author:** Brainstorming session with user
**Predecessor:** [v0.2 tools](2026-05-21-devtray-v0.2-design.md) (shipped as `v0.2.0` on 2026-05-21)

## Summary

v0.3 introduces persistence to DevTray. A new `DevTrayStorage` package backed by GRDB.swift adds the first SQLite database with one table — `tool_usage` — and a `UsageStore` protocol (declared in `DevTrayCore`) that `PopoverRoot` consumes via `@Environment`. Tool selection in the sidebar is automatically instrumented: every change to `selectedToolID` records a row, which feeds three clickable "frequent tools" chips rendered in the popover footer. Three known release-cycle papercuts are fixed along the way (dynamic version string, inline entitlements in `project.yml`, and a doc note about the `xcodebuild` SPM cache).

## Goals

- Land the first persistence layer with the minimum schema needed to unlock usage-based ranking.
- Keep `DevTrayUI` ignorant of GRDB — the UI depends only on the `UsageStore` protocol declared in `DevTrayCore`.
- Instrument tool selection automatically so any current or future tool is tracked without per-tool changes.
- Render the spec'd "frequent tools" footer chips, clickable to switch tool.
- Resolve recurring release-cycle papercuts touched by this slice.

## Non-goals

- No second table (snippets, FTS5) in v0.3 — both arrive in v0.5.
- No Settings UI to clear usage history or change the ranking window.
- No retention or pruning policy — `tool_usage` grows freely (estimated ~6 MB after 5 years of heavy use).
- No live re-ranking of chips within a single popover session — chips refresh once per popover open.
- No one-time UI notice on database corruption — `os.Logger` only in v0.3 (UI notice deferred to v0.4+).
- No Spotlight integration — v0.4 will *read* from this same store; no pre-emptive adjustments here.
- No snapshot/UI tests of `FrequentToolChip` — manual checklist suffices for v0.3.
- No rework of the per-tool `Package.swift` stub workaround — v0.3 adds no Tool package; rework lands with v0.5/v0.6 prep.

## Decisions (from brainstorming)

| # | Decision | Choice | Rationale |
|---|---|---|---|
| 1 | Persistence library | **GRDB.swift** (`^6.29`) | `DatabaseMigrator` is built-in, FTS5 ready for v0.5 snippets, safer Swift API (read/write blocks, value types) than raw SQLite3 — first SPM dep is justified |
| 2 | Schema in v0.3 | Single table `tool_usage(tool_id TEXT, used_at REAL)` + composite index | GRDB manages migration versioning internally — no separate `migrations` table needed; snippets/FTS5 deferred to v0.5 |
| 3 | Primary key on `tool_usage` | None (implicit `rowid`) + `INDEX (tool_id, used_at DESC)` | Spec's `PRIMARY KEY (tool_id, used_at)` would UNIQUE-fail on duplicate-millisecond clicks; rowid + index covers the only query (`COUNT GROUP BY tool_id WHERE used_at > X`) |
| 4 | Time storage | `Date.timeIntervalSince1970` as `REAL` | Matches the spec's existing `created_at` / `updated_at` REAL columns in the snippets schema (v0.5) — one convention across the database |
| 5 | Instrumentation | Auto via `PopoverRoot.onChange(of: selectedToolID)` | Single seam covers sidebar clicks, footer chip clicks, and future `Cmd+1…9` (v0.4); tools stay zero-touch and any new tool is instrumented automatically |
| 6 | Initial selection guard | `hasBootstrapped` State flag, set in `.onAppear` after seeding `selectedToolID` | The first nil→Tool assignment from popover open must not count as user intent; only post-bootstrap changes are recorded |
| 7 | Ranking window | `lastDays(30)` with fallback to all-time when fewer than 3 distinct tools used in window | Reacts to recent habit without the cold-start emptiness of a 7-day window; "always 3 chips after enough clicks" is the desired UX |
| 8 | Chip refresh cadence | Once per popover open, via `.task` on `PopoverRoot` | Live in-session re-ordering looks twitchy; the popover is dismissed and reopened many times per day — that's a natural refresh point |
| 9 | Footer layout | Chips left, version right, same 32pt height | Chosen via ASCII mockup comparison; preserves version visibility for support/debug without growing the popover |
| 10 | Dependency injection | `@Environment(\.usageStore)` with `InMemoryUsageStore` as `EnvironmentKey.defaultValue` | Protocol-based injection (not `@EnvironmentObject`); SwiftUI previews and `DevTrayUI` tests get a working store with zero config |
| 11 | `InMemoryUsageStore` location | `DevTrayCore` | Lets `DevTrayUI` (and any future consumer) exercise the protocol without taking GRDB as a transitive dependency |
| 12 | Boot failure handling | Fall back to `InMemoryUsageStore` and log via `os.Logger` if `SQLiteUsageStore.default()` throws | App stays functional under permission/disk errors; persistence resumes on next successful boot |

## Architecture

### Dependency direction

```
DevTrayApp (composition root)
    │  instantiates SQLiteUsageStore, injects via .environment(\.usageStore, …)
    ├──→ DevTrayCore       (protocols: UsageStore, ToolUsageRank, UsageWindow + InMemoryUsageStore + AppMetadata + EnvironmentKey)
    ├──→ DevTrayUI         (PopoverRoot consumes the protocol via @Environment)
    ├──→ DevTrayStorage    (SQLiteUsageStore, GRDB-backed; DatabaseMigrator; depends on DevTrayCore + GRDB.swift)
    └──→ Tools/*           (unchanged — tools do not know storage exists)
```

`DevTrayUI` gains **no** new dependency (still only `DevTrayCore`). `DevTrayStorage` is the only new package and the only one that pulls GRDB.

### New `DevTrayStorage` package

```
Packages/DevTrayStorage/
├── Package.swift                          (depends on ../DevTrayCore + GRDB ^6.29)
├── Sources/DevTrayStorage/
│   ├── DatabaseOpener.swift               (resolves URL, creates parent dir, corruption rename)
│   ├── Migrations.swift                   (DatabaseMigrator wiring + register all migrations)
│   └── SQLiteUsageStore.swift             (UsageStore implementation)
└── Tests/DevTrayStorageTests/
    ├── MigrationTests.swift
    └── SQLiteUsageStoreTests.swift
```

### New files in `DevTrayCore`

```
Packages/DevTrayCore/Sources/DevTrayCore/
├── UsageStore.swift              (protocol + UsageWindow enum + ToolUsageRank struct)
├── InMemoryUsageStore.swift      (test double + safe default — production-grade for one session)
├── UsageStoreEnvironment.swift   (EnvironmentKey + EnvironmentValues extension)
└── AppMetadata.swift             (version helper that reads Bundle.main.infoDictionary)
```

### Protocols (`DevTrayCore`)

```swift
public protocol UsageStore: Sendable {
    func record(toolID: ToolID, at date: Date) async
    func topTools(window: UsageWindow, limit: Int, now: Date) async throws -> [ToolUsageRank]
}

public enum UsageWindow: Sendable, Equatable {
    case lastDays(Int)
    case allTime
}

public struct ToolUsageRank: Hashable, Sendable {
    public let toolID: ToolID
    public let count: Int
}
```

`record` is `async` and **never** `throws` — failure is logged inside the implementation; losing the occasional write is acceptable, blocking the UI is not. `topTools` is `async throws` — callers use `try?` and degrade to an empty chip list on failure.

### Environment injection

```swift
// DevTrayCore/Sources/DevTrayCore/UsageStoreEnvironment.swift
private struct UsageStoreKey: EnvironmentKey {
    static let defaultValue: any UsageStore = InMemoryUsageStore()
}
public extension EnvironmentValues {
    var usageStore: any UsageStore {
        get { self[UsageStoreKey.self] }
        set { self[UsageStoreKey.self] = newValue }
    }
}
```

`PopoverRoot` reads `@Environment(\.usageStore)`. Tests and previews work without any wiring.

## Schema and migrations

### Migration v1

```swift
// DevTrayStorage/Sources/DevTrayStorage/Migrations.swift
import GRDB

enum Migrations {
    static func register(on migrator: inout DatabaseMigrator) {
        migrator.registerMigration("v1_tool_usage") { db in
            try db.execute(sql: """
                CREATE TABLE tool_usage (
                    tool_id TEXT NOT NULL,
                    used_at REAL NOT NULL
                );
            """)
            try db.execute(sql: """
                CREATE INDEX idx_tool_usage_tool_used
                ON tool_usage(tool_id, used_at DESC);
            """)
        }
    }
}
```

### Top-tools query

```sql
-- Window query (e.g. 30 days)
SELECT tool_id, COUNT(*) AS uses
FROM tool_usage
WHERE used_at > :cutoff
GROUP BY tool_id
ORDER BY uses DESC
LIMIT :limit;
```

If fewer than `limit` rows return, `SQLiteUsageStore.topTools` runs a second all-time query excluding already-returned `tool_id`s and concatenates until the limit is reached or no more data exists. The fallback lives in the implementation, not the UI.

### Database location

- File: `~/Library/Application Support/DevTray/devtray.sqlite`
- `DatabaseOpener` creates the parent directory if absent (idempotent).
- On `DatabaseQueue.init` failure: rename existing file to `devtray.sqlite.corrupted-<ISO8601>`, retry once, log via `os.Logger`. If retry also fails, the App composition root catches the throw and falls back to `InMemoryUsageStore` for the session.

### Retention

None in v0.3. The table grows unbounded. Estimated size: ~32 bytes/row × 100 clicks/day × 5 years ≈ 6 MB. Re-evaluate post-v1.0.

## Instrumentation contract

### `PopoverRoot` changes

```swift
@Environment(\.usageStore) private var usageStore
@State private var topTools: [ToolUsageRank] = []
@State private var hasBootstrapped = false

var body: some View {
    // … existing layout …
    .task { await refreshTopTools() }
    .onChange(of: selectedToolID) { _, newValue in
        guard hasBootstrapped, let id = newValue else { return }
        Task { await usageStore.record(toolID: id, at: .now) }
    }
    .onAppear {
        if selectedToolID == nil { selectedToolID = filteredTools.first?.id }
        hasBootstrapped = true
    }
}

private func refreshTopTools() async {
    let result = try? await usageStore.topTools(
        window: .lastDays(30), limit: 3, now: .now
    )
    topTools = result ?? []
}
```

### Why the `hasBootstrapped` guard

The `.onAppear` block performs the first `selectedToolID` assignment (nil → first tool). SwiftUI's `.onChange` fires on that assignment. Without the guard, every popover open would credit the default-selected tool — corrupting the ranking with "intent" the user never expressed. The guard ensures only post-bootstrap changes (real clicks) are recorded.

### Single seam, multiple sources

Every code path that mutates `selectedToolID` — sidebar tap, chip tap, and any future keyboard navigation (e.g. the `Cmd+1…9` shortcuts the main spec sketches) — automatically records usage. New tools added in v0.5/v0.6 require **zero** instrumentation work.

### App composition root

```swift
// App/DevTrayApp/DevTrayApp.swift
@StateObject private var registry = makeRegistry()
private let usageStore: any UsageStore = {
    do {
        return try SQLiteUsageStore.default()
    } catch {
        Logger(subsystem: "com.devtray.app", category: "storage")
            .error("SQLite open failed, using in-memory store: \(error.localizedDescription)")
        return InMemoryUsageStore()
    }
}()

MenuBarExtra("DevTray", systemImage: "wrench.adjustable") {
    PopoverRoot()
        .environmentObject(registry)
        .environment(\.usageStore, usageStore)
}
```

## UI changes

### `FrequentToolChip` (new component in `DevTrayUI`)

```swift
struct FrequentToolChip: View {
    let tool: AnyTool
    let action: () -> Void

    var body: some View {
        Button(action: action) {
            HStack(spacing: 4) {
                Image(systemName: tool.iconName).font(.caption2)
                Text(tool.displayName).font(.caption)
            }
            .padding(.horizontal, 8)
            .padding(.vertical, 3)
            .background(Capsule().fill(Color.secondary.opacity(0.15)))
        }
        .buttonStyle(.plain)
        .help("Frequent — \(tool.displayName)")
    }
}
```

### `PopoverRoot.footer` rewritten

```swift
private var footer: some View {
    HStack(spacing: 6) {
        ForEach(topTools, id: \.toolID) { rank in
            if let tool = registry.find(byID: rank.toolID) {
                FrequentToolChip(tool: tool) {
                    selectedToolID = tool.id   // triggers onChange → record
                }
            }
        }
        Spacer()
        Text("DevTray v\(AppMetadata.version)")
            .font(.caption)
            .foregroundStyle(.tertiary)
    }
    .padding(.horizontal, 12)
    .padding(.vertical, 6)
    .frame(height: 32)
}
```

### Empty state

When `topTools` is empty (fresh install, no clicks yet), `ForEach` emits nothing — the footer collapses to the version label on the right, with empty space on the left. No skeleton, no placeholder, no copy. Self-corrects on the first click.

### `AppMetadata.version`

```swift
// DevTrayCore/Sources/DevTrayCore/AppMetadata.swift
import Foundation

public enum AppMetadata {
    public static var version: String {
        Bundle.main.infoDictionary?["CFBundleShortVersionString"] as? String ?? "?"
    }
}
```

`AboutTab` in `App/DevTrayApp/DevTrayApp.swift` also switches to `Version \(AppMetadata.version)`. Future releases bump exactly **one** place — `CFBundleShortVersionString` in `project.yml`.

## Pre-work (gotchas closed by this slice)

### Gotcha #2 — XcodeGen clobbering entitlements

`bootstrap.sh` runs in v0.3 (new package enters `project.yml`). Without a fix, the third release in a row would strip `com.apple.security.network.client` from `DevTrayApp.entitlements`. Switch:

```yaml
# project.yml
entitlements:
  path: App/DevTrayApp/DevTrayApp.entitlements
  properties:
    com.apple.security.network.client: true
```

The entitlement is still unused at runtime (Sparkle ships in v0.7) but is permanently locked into source-of-truth.

### Gotcha #3 — Hardcoded version string

Folded into the UI work above. `AppMetadata.version` in `DevTrayCore`; one usage each in `PopoverRoot.footer` and `AboutTab`.

### Gotcha #4 — `xcodebuild` SPM cache

Comment header added to `scripts/test.sh`:

```bash
#!/usr/bin/env bash
# NOTE: xcodebuild may cache local SPM package state. If changes to
# Packages/* don't reflect after editing, prepend `clean` to the command:
#   xcodebuild clean build ...
# We do NOT clean by default (slow). See devtray-state memory for rationale.
```

No behavioral change. Documents the existing escape hatch.

### Deferred (not addressed in v0.3)

- **Gotcha #1 — per-tool `Package.swift` stub workaround.** v0.3 adds no new Tool package (only `DevTrayStorage`, which is storage-only). The architectural rework lands with v0.5 prep (Snippets) when the engine-before-view ordering bites again.

### Manual checklist (pre-merge, ~2 minutes)

1. Fresh boot, first launch — footer shows version only (no chips).
2. Click JWT 3×, JSON 5×, Hash 1×, quit, relaunch — footer shows chips `[JSON, JWT, Hash]` in that order.
3. Click the JSON chip — workspace opens JSON; relaunch confirms count incremented.
4. Version in footer and AboutTab both match `CFBundleShortVersionString` in `project.yml`.
5. `~/Library/Application Support/DevTray/devtray.sqlite` exists and is valid: `sqlite3 … '.schema tool_usage'` shows the expected table + index.
6. Rename file to `.sqlite.corrupted-test`, relaunch — app boots cleanly, new file created.

### CI

`scripts/test.sh` already iterates packages with a `testTarget` — `DevTrayStorage` is picked up automatically. GitHub Actions matrix unchanged.

## Exit criteria

- Test suite green: ~108–110 tests across all packages.
- All six manual checklist items pass.
- `CFBundleShortVersionString` set to `0.3.0` in `project.yml`.
- `grep -r "0\.2\.0" App Packages` returns no runtime literals (config and changelog are fine).
- `~/Library/Application Support/DevTray/devtray.sqlite` is created on first boot of the v0.3 build.
- PR opened from `feat/v0.3-storage` → `main`, reviewed, merged, annotated tag `v0.3.0` pushed.
- `devtray-state` memory updated: v0.3 status, new test count, new package, gotcha #1 still deferred.
