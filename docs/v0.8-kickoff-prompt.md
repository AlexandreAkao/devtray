## Antes de escrever spec/plano, faz brainstorm sobre (são as decisões reais deste ciclo):

1. **Conta de distribuição / método. ESTA É A DECISÃO QUE DESTRAVA TUDO.** Notarization exige **Apple Developer Program** ($99/ano) + um certificado **Developer ID Application**. Pergunta direta: eu tenho o Developer Program ativo? 
   - **Se sim** → caminho recomendado: **distribuição direta** (DMG/zip assinado+notarizado, hospedado no GitHub Releases), fora da Mac App Store.
   - **Se não** → ou a gente paga e segue, ou shippa só com ad-hoc + instrução "botão direito → Abrir" (UX ruim, mas viável pra um beta).
   - **Mac App Store** é a alternativa pesada: exigiria **sandboxing** + entitlements (mexe no fluxo de `NSOpenPanel`/`NSSavePanel` do export/import do v0.6, que hoje funciona por o app ser unsandboxed, e obriga reavaliar o hotkey global do `KeyboardShortcuts` e o Sparkle, que não roda dentro da MAS). Provável decisão: **NÃO MAS no v1**, distribuição direta. Confirmar.

2. **Sparkle: hospedagem do appcast + chaves + integração.** Sparkle 2.x via SPM (mais uma dep externa → esperar churn no `Package.resolved`, igual Yams/KeyboardShortcuts). Decidir: appcast em **GitHub Pages** (`gh-pages` ou `/docs`) vs asset no Release; gerar par de chaves EdDSA com o `generate_keys` do Sparkle (chave pública vai no `Info.plist` como `SUPublicEDKey`, privada vira secret do CI — **nunca commitada**); `SUFeedURL` apontando pro appcast; canal único "stable" pro v1. Item de menu "Check for Updates…" (na `MenuBarExtra` ou na cena `Settings`). Sparkle precisa do app **assinado** pra update seguro — então depende da perna 2.

3. **Release CI: gatilho, empacotamento e segredos.** Workflow novo (`.github/workflows/release.yml`) disparado em `push` de tag `v*`. Decidir: **DMG vs zip** (Sparkle aceita ambos; zip é mais simples, DMG é mais "produto"). O job precisa de **secrets do GitHub** que só EU consigo adicionar (o cert `.p12` em base64 + senha; credencial do `notarytool` — App Store Connect API key OU app-specific password; a chave privada EdDSA do Sparkle). O plano deve listar exatamente quais secrets criar, mas a criação é manual minha. **Atenção ao fluxo de tag:** hoje o padrão de release é "PR merge ff + tag anotada manual"; com release CI a **tag passa a DISPARAR o build** — decidir se a tag continua manual (eu crio depois do merge) ou se muda o ritual.

5. **Escopo + numbering.** Esse ciclo é grande e tem dependências externas (conta Apple, secrets) que travam partes. Decidir:

   - Se a conta Developer não estiver pronta, dá pra **fatiar**: fazer a perna do Sparkle + appcast + release CI com **assinatura ad-hoc** primeiro (já versiona e auto-atualiza num canal "beta"), e plugar Developer ID + notarization quando a conta sair. Decidir se fatia.

## Decisões de arquitetura/empacotamento (NÃO é brainstorm — segue o que já existe):

- **Versão mora em DOIS lugares:** `project.yml` (info.properties) E `App/DevTrayApp/Info.plist`. v0.8 → "0.8.0" / build "8" (ou "1.0.0"/"10" se a decisão 5 for ir direto pro 1.0). `AppMetadata.version` lê o `CFBundleShortVersionString`.

Bundle id `com.devtray.app`. Pra Developer ID: setar `DEVELOPMENT_TEAM` + provavelmente `CODE_SIGN_IDENTITY: "Developer ID Application"` no `project.yml` (ou só no step de export do CI, mantendo o dev local ad-hoc).
- **Dep externa nova (Sparkle):** declara no `project.yml` (`packages:` com `url:`/`from:`, igual KeyboardShortcuts/Yams) + linka no `DevTrayApp`. Esperar **churn no `Package.resolved`** (vaza pro primeiro leaf package — ver gotcha do v0.4/v0.7) — commitar.

## Lições que valem carregar (estão na memória, mas reforçando):

- **Bump de versão em DOIS lugares** (`project.yml` + `Info.plist`).
- **Dep externa → churn no `Package.resolved`** no primeiro leaf package; commitar pra reprodutibilidade (igual Yams no v0.7, KeyboardShortcuts no v0.4).

- **Padrão de release atual:** cada versão fecha com merge no `main` (PR, fast-forward) + tag anotada `v0.X.0` pushada; branch local deletada pós-merge, ref remota mantida. **Este ciclo pode mudar isso** (tag passa a disparar o CI de release) — decidir conscientemente.
- **SourceKit cross-module lag é ruído normal** — confia em `swift test` (por package), `./scripts/test.sh` (suite toda) e `xcodebuild -scheme DevTrayApp`. Clean Build Folder (`⌥⇧⌘K`) antes do primeiro smoke manual.
