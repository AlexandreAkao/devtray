Antes de escrever spec/plano, faz brainstorm rápido sobre:

1. **Global hotkey — `KeyboardShortcuts` (Sindre Sorhus) como o spec diz, ou `NSEvent.addGlobalMonitorForEvents` raw?** A lib é a 2ª dep SPM (depois do GRDB no v0.3) e traz uma UI pronta de captura pra Preferences. Raw é ~40 linhas de AppKit, zero dep extra, mas tem que construir o picker manual.

2. **Smart paste — onde mora a detecção?** Spec sugere `DevTrayCore/SmartPaste.swift` central. Mas a alternativa é estender o `Tool` protocol com `static func matches(clipboard: String) -> Bool` e deixar cada tool se autodescrever (JWT sabe "começa com `eyJ` + 2 pontos", Base64 sabe seu próprio padrão, etc.). Trade-off: central = um lugar pra debugar e ranquear; per-tool = extensível sem tocar Core mas espalha conhecimento.

3. **Ranking do Spotlight — quais sinais e como combinar?** Disponíveis: (a) frequência via `UsageStore.topTools(.allTime)`, (b) recência via `tool_usage.used_at` mais recente, (c) fuzzy match score na query digitada, (d) boost de smart-paste quando clipboard casa. Decisão: ranking simples (smart-paste sempre vence, depois frequência) ou weighted score? Spec deixou como open question.

4. **UX do smart-paste — auto-abrir ou só destacar?** Quando usuário invoca o hotkey com JWT no clipboard: (a) abre direto o JWT tool com o token pré-colado, (b) JWT aparece como top result + label "from clipboard", precisa ⏎ pra abrir, (c) banner "Paste as JWT?" acima da lista. Spec inclina pra (a)/(b) misto: ⏎ no top result abre com clipboard preloaded, ⌘⏎ vai pro popover.

**Decisão de empacotamento:** spec lista `DevTraySpotlight` como package separado. Vale confirmar no brainstorm se faz sentido manter separado (consistência com `DevTrayUI`) ou folder no `DevTrayUI` (pra v0.4 com escopo só Spotlight, talvez menos cerimônia). Decide isso na arquitetura, não como pergunta de brainstorm.
