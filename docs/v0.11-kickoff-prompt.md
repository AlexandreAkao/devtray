**O que ainda trava transformar em produto pago:** não tem **paywall**, não tem **license validation**, não tem **mecanismo de cobrança**. DevTray hoje é freeware bem polido. Este ciclo é a perna que monetiza — **EdDSA-signed license keys + LemonSqueezy como Merchant of Record (MoR)**. LemonSqueezy é a aposta porque (a) é MoR de verdade — eles cuidam de IVA/VAT/sales tax mundial, Apple Pay, Stripe, fraud, refund — você não precisa abrir empresa em N países; (b) tem webhook nativo pra `order_created` que dispara assinatura da license no seu backend; (c) flat fee de 5% + 50¢ por transação, muito mais barato que abrir uma S.A. e gerenciar tax compliance.

**Roadmap (memória reconciliada):**
- v0.11 (este) → license validation + paywall + LemonSqueezy. Destrava monetização.
- v1.0 (próximo) → launch polish (README/landing/ícone/DMG/anúncio público). v1 GA pago.
- Pós-v1.0 → dirigido por feedback de paying users; backlog frio inclui CloudKit sync, json jq-query, snippet change bus, bundle rename `DevTrayApp.app`→`DevTray.app`.

Bundle rename é candidato a fazer no v0.11 ou v1.0 — depois que tiver licenças amarradas a bundle ID, mudar de bundle ID fica caro. Levantar essa decisão no brainstorm.

## Antes de escrever spec/plano, faz brainstorm pra DECIDIR (decisões abertas — nada disso foi pré-discutido):

1. **Pricing model — one-time vs subscription?** Strawman: **one-time + paid upgrades a cada major version (à la Sublime Text/Tower)**. Aliña com expectativa de dev tool ("compra uma vez e fica meu"). Subscription rejeitado por (a) friction de cancelar, (b) precisaria SaaS-grade infra de billing recurring, (c) DevTray não tem custo marginal recorrente (sem sync, sem backend pesado). Counter: subscription paga mais long-term e força engagement com updates. **Confirmar/refutar.**

2. **Preço?** Strawman: **$19 one-time pra v1.x**, com upgrade fee opcional pra v2.x (talvez $9-12). Calibragem por benchmark: Tower $89, Paw $49, Charles $50, Postman free (mas SaaS), Insomnia free, RapidAPI $0-15/mo. DevTray escopo menor que Tower/Charles, maior que tool single-purpose. **$15? $19? $25? Confirmar.**

3. **Trial model — duração + hard expiry?** Strawman: **14 dias hard expiry, sem credit card**. 14 = padrão Sublime/JetBrains. Hard (não nag-only) porque sem hard expiry o trial dura pra sempre e nunca converte. Sem credit card upfront porque adiciona fricção brutal pra dev tool. Counter: 30 dias mais generoso; nag-mode permite usar 1x ano sem incomodar. **Confirmar.**

4. **Multi-machine — quantas activations por license?** Strawman: **3 machines, deactivable from server side**. 1 machine é frustrante (laptop pessoal + work + Mac mini caseiro é cenário comum dev), 5+ é convite a sharing. 3 funciona Sublime/Tower/Paw style. Deactivation via portal LemonSqueezy quando user troca máquina. Counter: ilimitado mesmo Apple ID (tipo Apple ID-bound). **Confirmar 3.**

5. **License key delivery — fluxo email→app?** Strawman: **LemonSqueezy webhook → backend assina license (EdDSA) → email user com a license key (HTTP-readable string) → user cola no Settings→License tab → app valida com pubkey embeddada e armazena no Keychain**. Alternativa fancier: deep link `devtray://activate?license=…` que abre o app direto. **Email + paste é mais robusto, confirmar.**

6. **Offline vs online validation?** Strawman: **offline-first via EdDSA, com revalidation online opcional pra revoke**. EdDSA = você assina no backend, app verifica com pubkey local, validação 100% offline. Online check só pra detectar revoke (chargeback, refund, deactivation). Tradeoff: revoke leva tempo a propagar (até next online check). Counter puro-offline: zero revoke, mas refund vira foot-gun. Strawman ataca o meio. **Confirmar offline + opcional online ping.**

7. **Backend hosting + framework?** Strawman: **Cloudflare Workers** (free tier 100k req/dia chega sobrando, deploy em segundos via Wrangler, latência global, sem cold start). Alternativas: Vercel functions, AWS Lambda, fly.io. CF Workers ganha por simplicidade. Linguagem: TypeScript (Wrangler default), com biblioteca EdDSA do `@noble/ed25519` (zero-deps puro JS). **Confirmar CF Workers + TS.**

8. **Domain / endpoint?** Strawman: **devtray.app** (compra agora, ~$15/ano, já serve pro launch landing também). Backend em `api.devtray.app/license/webhook`. Counter: subdomínio em domínio existente que já tenha. **Confirmar registro.**

9. **EdDSA keypair separate from Sparkle's?** Strawman: **sim, novo par dedicado pra licenses**. Sparkle key vive em GitHub Secret + tua keychain; license signing key vive em Cloudflare Worker secret (KV ou env var encriptada). Misturar é footgun se um vazar. **Confirmar separados.**

10. **Free tier após trial expira?** Strawman: **modo "read-only" — tools continuam funcionando, mas snippets bloqueados pra Save/Edit (read OK), e a maioria dos shortcuts globais idem**. Quem investiu já tem dado dentro, não merece perder. Counter: hard block total (mais conversão) ou full-free (não converte). Read-only é meio termo respeitoso. **Confirmar.**

11. **Machine identifier — como amarrar license a máquina específica?** Strawman: **hardware UUID via `IOPlatformUUID`** (`ioreg -rd1 -c IOPlatformExpertDevice | awk -F\" '/IOPlatformUUID/{print $4}'`). Persistente, hardware-bound, não muda em wipe/reinstall (a menos que swappe motherboard). Alternativas: volume UUID (muda se reinstala macOS), random UUID em UserDefaults (user pode resetar). **Confirmar `IOPlatformUUID`.**

12. **Activation flow — quando o app fala com o backend?** Strawman: **só na ativação inicial** (user cola license, app posta `POST /activate {license, machine_uuid}` → backend checa quota de activations, incrementa, retorna OK ou "too many"). Depois disso, app valida assinatura EdDSA local, nunca mais fala com server (a menos que user clique "Deactivate this Mac"). Minimal server load. **Confirmar.**

13. **Refund policy?** Strawman: **30-day no-questions-asked, padrão LemonSqueezy**. Refund dispara webhook `subscription_cancelled` ou `order_refunded` → backend marca license revoked. Próximo online check (ou na próxima ativação de máquina) o app detecta e cai pra trial-expired state. **Confirmar 30 dias.**

14. **Bundle rename `DevTrayApp.app` → `DevTray.app` — fazer agora ou nunca?** Sub-pergunta. Cosmético hoje; depois de v1.0 com licenças amarradas a bundle ID, mudar quebra todas as licenças instaladas. Strawman: **renomear neste ciclo** (mudança no `project.yml` `targetName`, rebuild). Aproveita que ainda não tem paying users instalados. Counter: pode quebrar Sparkle code-signing consistency check de novo (v0.8→v0.9→v0.10 testers já sofreram uma vez). **Confirmar/rejeitar.**

Strawman: **same repo, em `backend/`**. Mantém deploy+versionamento alinhado, sem repo gymnastics. CF Workers fica em `backend/license/`. Counter: separar dá CI/CD mais focado e secrets isolados. **Confirmar no brainstorm.**

- `backend/license/src/index.ts` — Worker entry. Routes:
  - `POST /webhook` (LemonSqueezy → assina license, manda email).
  - `POST /activate` (app → checa quota, incrementa contador, retorna OK).
  - `POST /deactivate` (app → decrementa, libera slot).
  - `GET /status` (app heartbeat opcional → retorna `active|revoked`).
- `backend/license/wrangler.toml` — config CF.
- `backend/license/package.json` — `@noble/ed25519`, `hono` (HTTP router minimal).
- **Cloudflare KV namespace** — `LICENSES`: key = license_uuid, value = `{ user_email, activations: [machine_uuid, ...], created_at, revoked: bool }`.
- **Secrets do Worker** (set via `wrangler secret put`): `LICENSE_PRIVATE_KEY`, `LEMONSQUEEZY_WEBHOOK_SECRET`, `RESEND_API_KEY` (ou alternativa email).

## Schema da license key (proposta — confirmar no brainstorm):

```
DT1-<base64url(payload).base64url(signature)>
```

Onde `payload` = JSON `{ "id": "uuid", "email": "user@x.com", "issued_at": 1234567890, "tier": "v1" }`. Sig = EdDSA(payload). Prefix `DT1-` identifica versão do schema (futuro DT2 pra v2.x upgrades).
